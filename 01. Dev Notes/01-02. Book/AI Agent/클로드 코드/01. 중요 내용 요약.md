### 1. 클로드 코드 입문과 개발 환경 구축

##### (1) 클로드 코드란
클로드 코드는 CLI상에서 클로드에 지시를 내리면서 특정 디렉터리 아래의 워크스페이스를 검색, 설명, 코딩, 계획하고 사용자의 애플리케이션 개발을 지원하는 AI 코딩 에이전트이다.

##### (2) 클로드 코드의 특징
- 멀티파일: 프로젝트 전체를 이해하고, 여러 파일을 한 번에 편집
- 병렬 실행 능력: 여러 작업을 동시에 처리하여 개발 속도가 비약적으로 향상
- 자연어를 통한 대화형 개발: 기술적인 세부 사항을 알지 못해도 자연어로 지시하면 구현 가능
- 실행 환경과의 깊을 통합: 터미널에서의 명령 실행, 테스트의 자동 실행, 배포까지 일관되게 지원
- 환경에 의존하지 않음: 터미널에서 가볍게 실행되므로 다양한 에디터 및 환경에 통합 가능

##### (3) 사용량에 영향을 미치는 요인
- 메시지 길이
- 첨부 파일의 크기
- 현재 대화의 길이
- 도구 사용(리서치, 웹 검색)
- 모델 선택
cf.) sonnet 모델 100만 토큰 분량의 컨텍스트 윈도 유지 가능(800~900페이지 책 분량)

### 2. MCP를 활용한 AI 챗봇 개발

##### (1) MCP 서버란
MCP 서버란 클로드 코드와 다른 애플리케이션 간 상호 작용을 중개하는 서버이다.
MCP 서버를 구축하고 이를 통해 데이터를 주고 받아 LLM이 특정 정보를 획득하거나 애플리케이션을 제어할 수 있게 된다. 구체적으로 Git 작업, 데이터베이스 작업, 노션과의 연동, API 연동 등 다양한 작업이 가능하며, 클로드 코드의 기능을 대폭 확장할 수 있다.
##### (2) 스코프 설정하기
- local
기본 설정이며, 현재 사용자만 클로드 코드를 실행한 디렉터리 하위에서만 사용한다.

- project
프로젝트 단위에서 MCP를 사용하고 설정을 공유할 때 사용한다.
.mcp.json에 설정이 기재되어, Git에서 관리될 경우 프로젝트의 전체 사용자에서 MCP 설정이 적용된다.

- user
사용자 홈 디렉터리의 ~/.claude에 설정이 저장되므로 해당 사용자의 모든 프로젝트에 적용된다.

- 앞으로 진행하는 프로젝트에서 이 MCP 서버를 항상 사용할 수 있게 되어 언제든지 Context7을 호출
```
claude mcp add context7 -s user -- npx -y @upstash/context7-mcp
```

##### (3) 추천 MCP 서버
###### 1) Context7
Context7은 라이브러리의 최신 공식 문서를 참조하는 MCP 서버로, 기술 스택을 선정할 때 더 정확한 정보를 제공한다. 이를 통해 클로드가 작성하는 코드에서 오류가 줄어들고, 라이브러리 명세에 정확히 맞는 코드를 생성할 수 있다.

- mcp 서버 설치는 해당 프로젝트에만 설치되므로 다른 프로젝틍데서 사용하려면 다시 claude mcp add를 실행해야 한다.

- 설치 방법
```
claude mcp add context7 -- npx -y @upstash/context7-mcp
```

- 프롬프트 예시
```
Todo 앱을 Next.js의 App Router를 사용해서 만들어주세요. 데이터는 로컬에 저장해주세요. use context7
```
이렇게 지시를 함으로써 Next.js의 최신 공식 문서의 기술이나 샘플 코드를 참고하여 만들어주므로 라이브러리 사용의 정확도가 높아진다.

- CLAUDE.md에 정리된 기술 선정 내용을 참고하여 지시
```
현재 구성 내용으로 애플리케이션을 만들어주세요. use context7
```

###### 2) Playwright
클로드 코드와 연결함으로써 LLM이 마치 눈을 뜬 것처럼 브라우저를 제어하게 하거나 프런트엔드의 작동과 오류를 확인할 수 있다.

- 프롬프트 예시
```
Chrome 브라우저에서 화면 너비가 450px 이하일 때, 3열로 표시되고 있는 "ItemList"가 화면을 벗어나고 있다. playwright MCP로 확인하여 수정해주세요.
```
```
개발 서버를 시작하고 playwright MCP 서버를 사용해서 localhost:3000에 접근하세요.
오류가 나지 않고 정상적으로 채팅할 수 있을 때까지 수정하세요.
```
브라우저의 콘솔 에러 등을 playwright MCP 서버를 통해 얻을 수 있어 백엔드에서 드러나지 않는 에러까지 잡아낼 수 있다.
###### 3) Serena
Serena는 코드의 구조와 문맥에 관한 이해를 클로드에 제공해 코드 생성 품질을 높일 수 있는 MCP이다.
Serena는 IDE처럼 LSP를 지원하여, 클로드가 코드를 더 깊이 이해할 수 있도록 돕는다.
Serena가 LSP의 분석 정보를 클로드에 전달하면, 클로드가 IDE와 같은 방식으로 프로젝트 내의 코드를 이해할 수 있게 된다.
참고로 Serena는 LSP를 프로젝트 단위로 실행해야 하므로, -s 옵션을 사용해 사용자 스코프로 설치할 수 없고 프로젝트마다 MCP 서버를 따로 설치해야 한다.

### 3. AI 챗봇 요구사항 정의

##### (1) 개발 환경 구축 및 요구사항 정의
- 프로젝트에 사용할 디렉터리 생성하고 이동한 후 프롬프트 작성
```
AI 챗봇을 만들고 싶습니다. 명세를 함께 작성하고 싶으니 필요한 정보를 질문해 줄수 있나요?
최종적으로 CLAUDE.md 파일을 출력하는 것이 목적입니다.
```

##### (2) 태스크 관리와 구현
###### 1) 실행 계획 수립과 TODO 리스트 작성
프로젝트에서는 대략적인 지시만으로 클로드가 적절히 작동할 수 있지만, <font color="#ff0000">작업 단위를 적절히 분할</font>하고 사전에 계획하면 사용자와 LLM 모두 작업을 더 쉽게 처리할 수 있다.
특히 하나의 지시로 계속 작업하면 클로드가 알아야 할 내용이 점점 많아지고, LLM이 전체 내용을 컨텍스트에 다 담지 못해 결과의 정확도가 떨어질 수 있다. 이로 인해 클로드가 일부 작업을 누락하거나 제대로 완료하지 못하는 경우도 발생한다.
따라서 실행 계획을 먼저 TODO 리스트 형태로 마크다운 파일에 정리하면, 클로드가 더 정확하게 작업을 수행할 수 있다. 또한 사용자는 각 단계의 생성 품질과 진행 상황을 쉽게 관리할 수 있다.

- 실행 계획 프롬프트 지시
프로젝트 루트에 있는 CLAUDE.md는 반드시 매번 컨텍스트에 포함되는 명세이므로 "이 애플리케이션", "현재의 애플리케이션 구성" 같은 지시 내용으로 전달된다.
```
현재 애플리케이션을 구축하기 위한 실행 계획을 세워주세요. 이를 TODO 리스트로 Markdown 파일에 정리하세요.
```

- TODO.md를 참조하도록 프롬프트에 지시
```
@TODO.md 이를 바탕으로 1단계를 구현하세요. 완료되면 체크박스에 체크하세요. use context7
```
	라이브러리를 읽어 들일 때는 use context7를 프롬프트에 추가

- 체크 박스를 활용한 진척 관리
```
완료되면 체크박스에 체크하세요.
```
이렇게 추가로 지시하지 않으면 하나의 TODO를 클로드 코드가 구현하는 것으로 끝나버러셔 어느 단계까지 구현했는지 기록할 수 없다. 세션이 길어지면 클로드가 기억하지 못할 수도 있으므로 이렇게 외부 메모리를 활용하는 편이 더 정확한 결과를 얻는 데 도움이 된다.

###### 2) 작업 내용을 돌아보게 하기
작업을 한 번 끝냈더라도 LLM이 모든 것을 구현하지 않았거나 코드의 일관성이 높지 않은 경우도 있다.
그러한 경우를 대비하여 원래 계획했던 모든 작업이 끝난 후 다음과 같이 전체 점검을 행하는 것이 바람직하다.
```
현재 코드에서 구현이 부족한 부분을 확인해서 남은 작업을 @TODO.md에 추가해주세요. ultrathink
```

### 4. 컨텍스트

##### (1) 컨텍스트 윈도
클로드 코드는 대화할 때마다 참조한 텍스트와 생성한 텍스트 전체를 계속 저장하여, 이를 문맥 이해와 추론에 활용한다. 이렇게 누적된 텍스트의 전체 분량을 컨텍스트 윈도라고 하며, 이는 모델의 "작업 기억" 역할을 한다.
컨텍스트가 커질수록 정보를 잊거나 실수할 가능성이 높아진다.
따라서 사람이 직접 컨텍스트를 관리하고, 일관성을 유지할 수 있도록 필요한 규칙과 제한을 마련해야 한다.
클로드는 컨텍스트 윈도를 최대 20만 토큰만큼 보유하도록 설계됐다.
대화뿐만 아니라 대화에서 참조해야 할 코드 정보 등도 보유하기 때문에 프로젝트 규모가 클수록 빠르게 제한에 도달한다. 토큰이 비대해지면 클로드 코드가 "무엇을 중요하게 여겨야 할지" 초점이 흐트러지고 응답의 정확도가 떨어진다.
##### (2) /context 실행 결과
컨텍스트 윈도우 구성 및 사용량 출력
###### 1) System prompt
###### 2) MCP tools
###### 3) System tools
###### 4) Memory tools
CLAUDE.md가 얼마나 차지하고 있는지를 보여줌.(~/.claude/CLAUDE.md, project/CLAUDE.md)
###### 5) Message (대화 이력)
###### 6) Skills(메타 데이터)
아래의 모든 요소가 합쳐져 컨텍스트 윈도우를 채운다.
##### (3) /compact
compact  명령은 지금까지의 대화 내용을 요약하고 압축한다. 따라서 기본적인 대화 흐름은 기억하지만, 내용이 간소화된다.

- 특정 내용에 초점을 맞추어서 압축
```
CLAUDE.md
# Summary instructions
대화 압축을 때는 코드의 변경 내용에 초점을 맞춰주세요.
```
클로드가 구현 내용에만 집중해서 기억할 수 있도록 설정하면 토큰 압축의 효율을 높일 수 있다.

##### (4) 가드레일의 예
- 컴파일 체크 언어 사용
- 테스트 파일을 작성
- 린트 검사
- DB 스키마의 타입 정의 출력
- API 스키마 정의
- 데이터 스키마 정의


### 5. 병렬처리와 서브 에이전트
##### (1) 깃 워크트리를 활용한 병렬 처리
깃 워크트리는 동일한 깃 레포지터리의 다른 브랜치를 별도 디렉터리에 체크아웃할 수 있는 기능이다.
이를 통해 원래 작업 중인 브랜치의 코드에 영향을 주지 않으면서 병렬로 작업할 수 있다.
앤트로픽 공식 문서에서도 깃 워크트리를 이용한 병렬 처리를 권장한다.
git worktree를 사용하면 추가 복제 없이 하나의 .git 디렉터리를 공유하며 여러 작업 환경을 만들 수 있다.

```
git worktree add -b feature-x ./feature-x
git worktree remove feature-x
```

|부분|설명|
|---|---|
|`git worktree add`|새로운 worktree를 추가하는 명령|
|`-b feature-x`|`feature-x`라는 **새 브랜치를 생성**|
|`./feature-x`|worktree가 체크아웃될 **디렉토리 경로**|

실행 결과)
```
your-repo/          ← 메인 worktree (현재 브랜치)
your-repo/feature-x ← 새로 생성된 worktree (feature-x 브랜치)
```
두 디렉토리가 **같은 `.git`을 공유**하면서 서로 다른 브랜치를 독립적으로 체크아웃한 상태가 됩니다.

##### (2) 병렬 처리 팁
다른 세션의 클로드는 별개의 인격과 같으므로 서로의 작업이 겹치지 않도록 하는 것이 기본적인 팁이다.
따라서 프런트엔드, 벡엔드, 테스트, 인프라 등 각각 담당할 작업을 나누는 것이 좋다.
##### (3)  서브 에이전트 활용
서브 에이전트는 서로 컨텍스트를 공유하지 않기 때문에, 사용자가 매번 컨텍스트를 전환할 필요없이 서브 에이전트가 각 전문 영역의 작업에 집중하도록 할 수 있다.
서브 에이전트는 호출될 때 마다 초기화된 상태로 시작된다. 따라서 작업을 수행하는 데 필요한 정보를 다시 수집하는 데 시간이 걸려서 처리가 느려질 수 있다.

- 서브 에이전트 작성법(대화모드)
서브 에이전트는 설정 파일을 통해서도 만들수 있고, 대화모드에서 바로 생성할 수 도 있다. 대화 모드에서 바로 생성하는 것이 더 쉽고 편리하므로 이를 권장한다.
```
/agents
--> Create new agent
1. Project(.claude/agents): 이번 프로젝트에 참여하는 모든 멤버가 서브에이전트를 함께 사용 가능
2. Personal(~/.claude/agents): 다른 멤버 공유 없이 개인만 사용할 수 있는 서브에이전트 생성
--> Creation method
3. Generate with Claude: 클로드에게 요청하면 좀 더 세밀하게 서브에이전트가 정의
```

### 6. 커스텀 슬래시
클로드 코드에 대한 프롬프트를 명령어로 만들 수 있다.
커스텀 슬래시 명령을 통해 여러 개의 지시를 하나로 묶어 슬래시 명령으로 미리 정의해둘 수 있다.
자주 반복하는 지시를 할 때 사용하면 유용하다.

- 커스텀 슬래시 명령 만드는 방법
커스텀 슬래시 명령은 project(~)/.claude/commands/CUSTOM_HOOK_COMMAND.md 파일을 생성하여 만든다.
- 커스텀 슬래시 명령에 인자 전달하기
커스텀 슬래시 명령에는 인자를 전달할 수 있으며, 프롬프트 내에서 $ARUMENTS라고 기술하면 명령을 호출할 때 특정 값을 제공할 수 있다.
ex.)
pre-review 라는 슬래시 명령을 만들고 마크다운 파일에 다음과 같이 기재해둘 수 있다.
```markdown
풀 리퀘스트 #$ARUMENTS를 검토하세요.
```

```terminal
/pre-review 14
```

### 7. 훅
클로드 코드의 다양한 처리 시점에 맞춰 특정 명령어를 실행할 수 있다.

- 훅을 만드는 방법
(~)project/.claude/settings.json 안에 실행 타이밍(후킹 이벤트)과 실행하고 싶은 스크립트를 작성하면 된다.
이때 실행 타이밍은 클로드 코드 세션 시작 시점이나 종료 시점, 명령어 사용 시점 등이다.

- 클로드가 파일을 새로 생성하거나 편집할 때 포매터나 린터를 실행하는 훅 설정
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|MultiEdit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"$CLAUDE_TOOL_INPUT\" | jq -r '.file_path // .path // \"\"' | grep -E '\\.(ts|tsx|js|jsx|mjs|cjs)$' | xargs -I{} sh -c 'npx prettier --write \"{}\" 2>&1 && echo \"✅ Prettier: {}\"'",
            "timeout": 30,
            "statusMessage": "Prettier 포맷 중..."
          },
          {
            "type": "command",
            "command": "echo \"$CLAUDE_TOOL_INPUT\" | jq -r '.file_path // .path // \"\"' | grep -E '\\.(ts|tsx|js|jsx|mjs|cjs)$' | xargs -I{} sh -c 'npx eslint --fix \"{}\" 2>&1; exit 0'",
            "timeout": 60,
            "statusMessage": "ESLint 검사 중..."
          }
        ]
      }
    ]
  }
}
```

- 훅 테스트하기
```
@settings.json이 훅이 작동하는지 테스트하세요.
```

- 훅의 장점
확률적으로 정해지는 클로드의 출력 결과에 대해 확실하게(결정론적으로) 실행이나 출력 결과를 가져온다는 것이다. 따라서 클로드의 실행에 확실성을 부여하고 싶을 때는 훅을 적극적으로 활용하면 좋다.

- 훅 활용 예
1) 비밀 파일에 대한 접근을 차단하거나 설정 파일의 삭제를 방지할 수 있다.
2) 프롬프트에 특정 컨텍스트를 포함시켜 클로드가 기억해야 할 내용을 항상 상기 시킬 수 있다.
3) 클로드의 알림을 데스크탑이나 슬랙에 보내 대기 시간을 효율적으로 활용할 수 있다.

### 8. 에이전트 스킬
에이전트 스킬을 이용하면 에이전트에 특정한 전문 스킬을 부여할 수 있다.
예를 들어, 파워포인트나 엑셀 파일 같은 문서 생성, 데이터로부터의 금융 또는 비즈니스 분석, 또는 특정 Git 작업이나 작업 자동화 등  다양한 능력을 부여할 수 있다.

- 에이전트 스킬 도입 이유
원하는 기능을 간단하고 명확하게 지시하면 컨텍스트 관리가 더 효율적이고 에이전트의 성능도 향상될 수 있다.
에이전트 스킬을 활용하면 LLM에 구체적인 절차, 스크립트, 참조 파일 등을 제공할 수 있어 프롬프트나 훅만으로는 어려웠던 특정 작업의 재현성과 일관성을 보장할 수 있다.
사용자의 요청과 스킬이 존재하면 클로드가 그 스키을 우선적으로 활용한다. 또한 사용자가 스킬을 지정해 직접 호출할 수도 있다.

- 스킬 설정 파일
스킬 설정 파일은 특정 프롬프트와 프로그램 스크립트로 이뤄져 있다. 쉽게 말해, 에이전트가 따라야 할 절차를 안내하는 메뉴얼과 이 때 실행할 프로그램을 함께 제공하는 것이다.

```
my-skill
	SKILL.md(필수)
	reference.md(옵션: 컨텍스트에 포함하고 싶은 문서)
	example.md(옵션: 구페적인 예시)
	scripts/
		helper.py(옵션: 실행할 스크립트)
	templates/
		template.txt(옵션: 출력 템플릿 등)
```

- 에이전트에게 스킬 생성 요청 지시
```
새로운 스킬을 작성하세요.
클로드 코드 에이전트가 파일 편집을 완료한 후, 편집된 파일에 대해 린터를 실행하는 스킬이다.
```
이렇게 하면 예를 들어 ~/.claude/skiils/lint-checker라는 디렉터링 다음과 같은 SKILL.md가 출력된다.

- 스킬의 구체적인 사용법
구체적인 절차나 명령을 지정하고, 스크립트의 결과를 클로드가 컨텍스트로 포함하기 때문에 완료된 시점에 예상되는 출력이나 오류 발생 시에 지정하는 것도 가능하다.
예를 들어, git diff에서 추출한 파일로 특정 형식의 커밋 메시지를 만들거나, 품질 관리 기준에 따라 코드 리뷰를 진행하거나, 코드를 작성할 때 특정 예시에 기반한 테스트 파일을 반드시 작성하도록 하는 등 여러 상황에 적용할 수 있다.
이는 LLM의 높은 유연성과 처리 능력에 스크립트 실행 기능이 더해져 매우 강력하게 활용할 수 있다.
