
-  RAG 기초: 검색 → 증강 → 생성
RAG 시스템은 사용자 질문에 답변하기 위해 LLM에만 의존하는 것이 아니라 외부 지식을 참고하도록 만든다.
<font color="#ff0000">사용자 질문에 대해 대답할 수 있는 벡터 DB의 유사 문서를 검색하고, 이를 바탕으로 LLM이 답변하는 원리이다.</font>
먼저 사용자의 질문을 받으면, 이와 유사한 정보를 벡터 DB에서 검색한다.(검색된 청크는 사용자의 질문과 합쳐져 LLM에게 전달할 최종 프롬프트가 완성된다.)
그리고 사용자 질문과 검색된 유사 정보를 하나의 프롬프트로 담아 LLM에 전달하여 올바른 답변을 생성한다.
학생이 모르는 문제에 대해 교과서를 참고하여 답변을 하듯이, LLM에게도 참고할 수 있는 자료를 주고 이를 기반으로 답변하도록 만든다. 따라서 사용자가 질문에 특정 자료들을 통해 답변할 수 있다면, 이를 모두 하나의 데이터베이스로 구축하고 참고 자료를 검색하여 올바른 답변을 내도록 만들 수 있다. 결과적으로 LLM은 학습할 때 형성된 자체 지식에만 의존하는 것이 아니라, 외부 지식과 결합하여 답변을 생성할 수 있게 되어 환각 현상을 크게 낮출 수 있다.

- RAG 시스템 구축에 필요한 구성 요소
	LLM 모델
	Document Loader
	Text Splitter
	임베딩 모델
	Vector Store
	Retriever

- RAG의 본질: LLM이 올바른 정보 출처를 찾아 이를 기반으로 환각 현상을 해결하는 것이다.
LLM에게 아무런 맥락을 주어지지 않은 프롬프트를 입력하면 사전 학습시 습득한 지식을 바탕으로 답변을 생성하는데, 이때 사전 학습에서 배우지 못한 개념이나 사실들은 환각 현상을 일으키기 쉽다. RAG에서는 증강 단계에서 주어진 프롬프트를 바탕으로 LLM이 답변을 생성한다.
이렇게 프롬프트 안에 맥락을 주입하면, 기존에 학습되지 않은 사실이나 지식에 대해서 LLM이 이해하고 알맞은 답변을 제공할 수 있다.

- 파인튜닝
학습 데이터셋 구축, 모델 파인튜닝 작업이 필요
파인튜닝을 하기 위해서는 질문-답 형태의 데이터셋을 구성해야하므로 시간이 많이 필요하다.
RAG의 경우 주어진 문서를 얼마나 잘 임베딩하고, 유사 청크를 얼마나 검색 및 추출하는지에 따라서 성능이 결정된다. 파인튜닝의 경우 데이터셋의 품질과 적용 방법론에 따라서 성능이 달라진다.

- 파인튜닝 선택지
오픈 소스 LLM으로만 서비스를 운영할 계획이고 시간과 비용이 충분하다면, 특화시킬 도메인의 데이터셋을 기반으로 파인튜닝을 진행하는 것이 좋다. 또한 구현할 서비스가 일반적인 LLM과 다른 형태의 답변을 필요로 한다면 파인튜닝을 하는 것이 좋은 선택지이다.

- 랭체인
오픈 소스 프로젝트로 LLM 앱 구축을 위한 프레임워크이다.
랭체인에는 프롬프트 변형을 위한 프롬프트 템플릿뿐만 아니라, 더욱 광범위한 LLM 서비스 개발을 위한 다양한 모듈을 제공한다. 핵심 구성 요소는 주로 RAG를 구축하기 위한 모듈로, 여러 LLM을 애플리케이션에 통합할 수 있는 Models, 사용자의 프롬프트를 재가공하는 Prompts, 벡터 DB로 구출할 문서를 불러오는 Document Loaders, 이를 여러 청크하는 Text Splitters, 분할된 텍스트 청크들을 저장하는 Vector Stores, 원하는 답변의 형태로 재가공하는 Output Parsers 등이 있다. 
> ★ 문서 로딩 → 분할 → 임베딩 → 검색 → 답변 생성
> - 체인: LLM과 다양한 모듈을 하나의 패키지로 묶어주는 모듈

- Text Splitters
	Text Splitter의 역할은 Document를 특정 기준에 따라 정해진 길이의 청크로 분할한다. 만약 하나의 문서가 사용자가 정의한 길이보다 길면, 이를 Text Splitter로 분할하여 여러 개의 청크로 저장하는 것이다. 이렇게 여러 개의 청크로 분할된 여러 문서를 벡터 DB에 담는다. 그러면 사용자 질문과 벡터 DB 내 모든 청크 간의 유사도를 계산하여 유사도가 높은 N개의 청크를 검색할 수 있게 된다.
	1) CharacterTextSplitter
	2) RecursiveCharacterTextSplitter: 여러 개의 구분자를 재귀적으로 적용하여 텍스트를 분할 (CharacterTextSplitter에서 발생하는 청크 사이즈 초과 문제를 해결. 즉, chunk_size 초과하는 경우 두번째 구분자를 적용하여 텍스트를 재분할하여 사용자가 지정한 chunk_size를 절대 초과하지 않도록 만듬)
	3) SemanticChunker: 같은 맥락의 문장은 하나의 청크로 담고 맥락이 달라지는 경우 다른 청크로 분할. 즉, 목적은 문장 간 유사성이 떨어지는 구간에서 분할 하는 것이다.

- Vector Stores
벡터 DB는 여러 문서의 청크를 임베딩하여 얻어낸 임베딩 벡터의 데이터베이스이다.
벡터 DB는 고차원의 벡터 데이터를 효율적으로 저장하고 검색할 수 있도록 설계되었다. 텍스트나 이미지와 같은 비정형 데이터를 임베딩 벡터로 변환하여 벡터 DB에 저장하면 유사도 검색을 통해 쉽게 정보를 조회할 수 있다.
벡터 DB에서는 ANN(Appromimate Nearest Neighbor) 알고리즘을 바탕으로 임베딩 간 유사도를 계산하여 검색에 활용한다.

| 구분                    | 벡터 DB 종류                         |
| --------------------- | -------------------------------- |
| 순수 벡터 데이터베이스          | Chroma, Weaviate, Qdrant         |
| 텍스트 전용 데이터베이스         | Elasticsearch, OpenSearch, Solr  |
| 벡터 라이브러리              | FAISS, Annoy, Hnswlib            |
| 벡터 기능 추가된 NoSQL       | MongoDB, neo4j, redis            |
| 벡터 저장 및 검색 가능한 SQL DB | PostgreSQL, Clickhouse, Kinetica |
	 ChromaDB
	 RAG 구축 시 가장 많이 활용되는 오픈 소스 벡터 DB이다. 벡터 임베딩을 다루기 위한 대표적인 기능으로 임베딩과 메타데이터 저장, 문서 및 질문 임베딩, 임베딩 검색 등이 있다.

- Retriever
사용자 질문과 답변 근거가 될 문서의 연결하는 것이 굉장히 중요하다.
1) 사용자의 질문을 어떻게 해석할 것인가
2) 답변 근거가 될 문서를 어떻게 얼마나 가져올 것인가
RAG의 검색 단계는 많은 고려 사항을 검토하여 최적의 파라미터를 찾아내야 하는 단계이다.
랭체인에서는 이 과정을 보조하는 다양한 모듈들을 갖고 있으며, 이를 Retriever라고 부른다.
랭체인의 Retriever 모듈을 통해 백터 DB를 결합하면 더 쉽고 세밀한 검색 기능을 만들 수 있다.

- MultiQueryRetriever
질문 문장이 벡터 DB 내 청크들과 유사하지 않은 경우 적절한 근거를 찾기 어렵다.
이를 해결하기 위한 방법은 사용자 질문을 여러 버전으로 만들어 벡터 DB 검색이 원활하게끔 하는 것이 바로 MultiQueryRetriever이다. 사용자 질문의 목적을 다각도로 해석하여 여러 버전의 질문 문장을 만들어내는 것이 핵심이다. MultiQueryRetriever는 원본 질문에 대한 여러 버전의 질문을 생성하기 위해 LLM을 따로 설정해줘야 하는 특징이 있다. LLM은 Retriever에 주어진 질문의 의도를 파악하고, 여러 버전의 질문을 생성하여 검색 결과가 더욱 풍부하도록 촉진시킨다.

- MultiVectorRetriever
문서의 벡터를 재가공항 검색 품질을 향상시킨다.
청크의 길이를 정하는 것이 답변의 품질을 좌우한다. 상위 문서 검색기는 청크를 나누는 기준을 2개로 정의할 수 있다. 상위 청크 기준으로 긴 길이의 청크를 만들고, 그 하위 청크 기준으로 짧은 길이의 청크를 만드는 것이다. 이렇게 하면 사용자의 질문과 유사한 문서를 검색할 때, 짧은 길이의 하위 청크를 검색하여 더욱 정확하게 추출할 수 있으며, LLM에게 컨텍스트를 전달할 때 상위 청크를 전달하여 조금 더 완전한 맥락을 주입한 답변을 풍부하게 할  있다.

- LCEL(LangChain Expression Language)
랭체인에서는 여러 모듈을 엮어 하나의 파이프라인 즉, 체인을 만드는 것이 핵심인데 LCEL은 이 과정을 간단하게 만들어준다. 또한 모듈을 각각 구성한 후 파이프 오퍼레이터를 기반으로 체인을 빠르게 만들 수 있다.
기본적으로 체인은 '프롬프트 템플릿 -> LLM(모델) -> 출력 파서' 구성의 3단 결합이 일반적이다.
→ '|'(파이프 오퍼레이터) 기호를 통해 모듈을 연결함으로써 구문의 복잡성을 최대한 제거하였다.
	RunnablePassthrough: LCEL이 잘 구동하도록 랭체인에서 개발한 고유한 객체 Runnable의 일종이다.
	Runnable은 LCEL에서 '|'을 기준으로 여러 구성 요소를 Chain으로 묶을 때 더 많은 조정이 가능하도록 돕는다.
```python
#LCEL로 프롬프트 템플릿 - LLM - 출력 파서 연결하기
chain = prompt | model | output_parser
```

- LangChain Hub
랭체인에서 추가한 플랫폼으로 Langsmith라는 랭체인 관리 플랫폼의 일부이다. 이 플랫폼의 목적은 프롬프트를 공유하고 관리하는 것으로, 랭체인 활용 시 참고할 만한 프롬프트가 다수 존재한다. RAG에 필요한 프롬프트도 랭체인 Hub에서 찾을 수 있으며 파이썬 코드로 이 프롬프트를 호출하여 활용할 수도 있다.
