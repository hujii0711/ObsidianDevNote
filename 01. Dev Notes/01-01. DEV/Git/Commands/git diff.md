`git diff`는 한마디로 **"두 지점 사이의 차이점(Difference)을 보여주는 명령어"**입니다. 코드의 어떤 줄이 추가되었고, 어떤 부분이 삭제되었는지 줄 단위로 대조하여 시각적으로 보여주는 역할을 합니다.

---

### 1. 기본적인 개념

`git diff`를 실행하면 터미널에 다음과 같은 형식으로 결과가 출력됩니다.

- **`+` (초록색 줄):** 새로 추가된 코드
    
- **`-` (빨간색 줄):** 삭제된 코드
    

기본적으로 `git diff`는 세 가지 영역을 비교하는 데 사용됩니다.

1. **Working Directory**: 현재 내가 수정 중인 파일
    
2. **Staging Area (Index)**: `git add`를 통해 올라간 파일
    
3. **Repository (Commit)**: 이미 저장된 커밋 기록
    

---

### 2. 어떤 상황에서 유용한가?

#### ① 커밋하기 전 마지막 검토 (`git add` 전)

가장 많이 쓰이는 상황입니다. 파일을 수정하고 `git add`를 하기 전에, 내가 정확히 어떤 내용을 고쳤는지 스스로 리뷰할 때 사용합니다. 의도치 않게 들어간 테스트 코드나 오타를 잡아낼 수 있습니다.

> `git diff` (작업 디렉토리 vs 스테이징 영역)

#### ② 스테이징된 내용 확인 (`git add` 후)

이미 `git add`를 눌러 스테이징 영역에 올린 파일들이 무엇인지, 그 내용이 무엇인지 확인하고 싶을 때 사용합니다.

> `git diff --cached` 또는 `git diff --staged`

#### ③ 커밋 간의 차이 비교

어제 한 커밋과 오늘 한 커밋 사이에서 무엇이 바뀌었는지, 혹은 특정 파일의 과거 버전을 확인하고 싶을 때 유용합니다.

> `git diff [커밋ID_A] [커밋ID_B]`

#### ④ 브랜치 간의 차이 확인 (협업의 핵심)

내 브랜치와 동료의 브랜치, 혹은 `main` 브랜치와 내 작업 브랜치를 비교할 때 필수적입니다. 앞서 배운 `fetch` 이후에 충돌 여부를 가늠할 때도 이 방식을 씁니다.

> `git diff main feature/login` (main 브랜치와 login 브랜치 비교)

#### ⑤ 파일 이름만 빠르게 확인

수정된 내용이 너무 많아 복잡할 때는, 어떤 파일들이 바뀌었는지만 목록으로 뽑아볼 수 있습니다.

> `git diff --name-only`

### 💡 팁: 더 직관적으로 보는 방법

터미널에서 텍스트로 보는 것이 불편하다면, VS Code 같은 IDE의 **'소스 제어'** 탭을 활용해 보세요. 왼쪽에는 이전 코드, 오른쪽에는 수정된 코드가 나란히 배치되어 훨씬 읽기 편합니다.

`git diff`는 단순히 "무엇이 바뀌었나"를 보는 단계를 넘어, **"내가 의도한 대로 코딩했는가"를 검증하는 가장 첫 번째 방어선**입니다. 일요일 밤 마무리 작업하시면서 `git diff`로 오늘 코드를 한번 쭉 훑어보시는 건 어떨까요?
