`git reset`은 한마디로 **"과거로 시간을 되돌리는 명령어"**입니다. 현재 브랜치의 끝(HEAD)을 특정 커밋 지점으로 강제로 이동시켜, 그 이후에 발생한 작업들을 취소하거나 상태를 변경할 때 사용합니다.

`reset`은 되돌리는 강도에 따라 세 가지 모드로 나뉩니다. 이 차이를 아는 것이 핵심입니다.

---

### 1. 세 가지 주요 모드

1. **`--soft` (가장 약함)**
    
    - **행동:** 커밋만 취소하고, 파일 수정 내역은 그대로 **스테이징 영역(Staging Area)**에 남겨둡니다.
        
    - **결과:** 바로 다시 `git commit`을 할 수 있는 상태가 됩니다.
        
2. **`--mixed` (기본값)**
    
    - **행동:** 커밋도 취소하고, 스테이징도 취소하지만 **파일 수정 내역(Working Directory)**은 보존합니다.
        
    - **결과:** 파일은 수정된 상태로 남아있지만, 다시 `git add`부터 해야 하는 상태입니다.
        
3. **`--hard` (가장 강함 - 주의!)**
    
    - **행동:** 커밋, 스테이징, 파일 수정 내역을 **모두 삭제**하고 지정한 커밋 상태로 완전히 되돌립니다.
        
    - **결과:** 해당 커밋 이후의 모든 작업이 사라집니다. (복구가 어려우니 신중해야 합니다.)
        

---

### 2. 어떤 상황에서 유용한가?

#### ① 방금 한 커밋의 메시지나 내용을 수정하고 싶을 때

커밋을 했는데 오타가 있거나 파일을 하나 빼먹었다면, `--soft` 리셋으로 커밋만 살짝 취소한 뒤 다시 수정해서 커밋할 수 있습니다.

> `git reset --soft HEAD~1` (직전 커밋 취소)

#### ② `git add`를 실수로 너무 많이 했을 때

커밋할 생각이 없는 파일까지 스테이징 영역에 올렸다면, `--mixed` 리셋을 통해 스테이징만 초기화할 수 있습니다.

> `git reset HEAD` (현재 스테이징된 파일들 내리기)

#### ③ 작업 내용이 완전히 꼬여서 처음부터 다시 하고 싶을 때

이것저것 코드를 건드리다가 도저히 수습이 안 될 때, 마지막 커밋 상태로 깔끔하게 돌아가기 위해 사용합니다.

> `git reset --hard HEAD` (마지막 커밋 상태로 강제 초기화)

#### ④ 잘못된 브랜치에 커밋했을 때

`main` 브랜치에 직접 커밋하면 안 되는데 실수로 커밋했다면, `--soft` 리셋으로 작업 내용은 남긴 채 브랜치만 옮겨서 다시 커밋할 수 있습니다.

---

### ⚠️ 주의사항

`rebase`와 마찬가지로, **이미 원격(GitHub)에 Push된 커밋에 대해서는 `reset`을 사용하지 않는 것이 원칙**입니다. 협업 중인 다른 사람들의 이력과 충돌이 발생하기 때문입니다.

- **로컬 작업 중:** `git reset` (이력을 지워버림)
    
- **이미 Push한 후:** `git revert` (기존 이력은 남겨두고 '반대되는 커밋'을 추가해서 되돌림)
    

### 💡 요약

- `--soft`: 커밋만 취소 (내용은 그대로)
    
- `--mixed`: 스테이징까지 취소 (내용은 그대로)
    
- `--hard`: 다 취소 (내용까지 삭제)
