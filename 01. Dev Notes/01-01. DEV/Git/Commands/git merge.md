`git merge`는 한마디로 **"갈라진 두 갈래의 역사를 하나로 합치는 작업"**입니다. 독립된 브랜치에서 작업한 내용을 메인 줄기(보통 `main` 또는 `master`)나 다른 브랜치에 통합할 때 사용합니다.

---

### 1. 기본적인 개념

`merge`를 실행하면 Git은 두 브랜치의 공통 조상(Base)으로부터 각각 어떤 변화가 있었는지 계산하여 하나로 합칩니다. 방식은 크게 두 가지로 나뉩니다.

- **Fast-Forward**: 합치려는 브랜치가 대상 브랜치로부터 단순히 앞서가기만 한 상태일 때, 별도의 커밋 없이 포인터만 앞으로 이동시킵니다.
    
- **3-Way Merge**: 두 브랜치가 서로 다른 방향으로 진행되었을 때, 두 내용을 합친 새로운 **'머지 커밋(Merge Commit)'**을 자동으로 생성합니다.
    

---

### 2. 어떤 상황에서 유용한가?

#### ① 기능 개발 완료 후 통합 (Feature Workflow)

새로운 기능(로그인, 결제 시스템 등)을 별도의 `feature` 브랜치에서 개발하다가, 테스트가 끝나고 완성되었을 때 메인 코드베이스에 합치기 위해 사용합니다. 협업의 가장 표준적인 과정입니다.

#### ② 다른 사람의 작업물 가져오기

협업 중 동료가 업데이트한 최신 코드를 내 작업 브랜치에 반영해야 할 때 사용합니다. 앞서 배운 `git pull`이 내부적으로 이 `merge`를 수행하는 것입니다.

#### ③ 버그 수정 반영 (Hotfix)

배포된 서비스에서 긴급한 버그를 수정하여 `main`에 반영한 뒤, 현재 개발 중인 다른 브랜치들에도 그 수정 사항을 똑같이 적용해야 할 때 유용합니다.

---

### 3. 머지 충돌 (Merge Conflict)

`merge`를 할 때 가장 긴장되는 순간은 **충돌**입니다. 같은 파일의 같은 줄을 서로 다르게 수정했을 때 발생합니다.

- Git은 어느 쪽 코드가 맞는지 스스로 판단하지 못하므로, 사용자에게 직접 선택하라고 알려줍니다.
    
- 이때 개발자는 충돌이 난 부분을 직접 확인하고 수정 후 다시 커밋해야 합니다.
    

---

### 💡 팁: Rebase와 무엇이 다른가요?

- **`merge`**: "우리가 언제 합쳐졌는지"에 대한 기록(머지 커밋)을 남깁니다. 역사가 복잡해질 수 있지만, 실제 작업 흐름을 정확히 보존합니다.
    
- **`rebase`**: 역사를 한 줄로 깔끔하게 만듭니다. 기록을 예쁘게 다듬고 싶을 때 쓰지만, 실제 합쳐진 시점의 기록은 사라집니다.
