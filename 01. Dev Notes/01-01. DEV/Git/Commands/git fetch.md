`git fetch`는 원격 저장소(Remote Repository)의 최신 이력을 내 로컬 저장소로 **'가져만 오는'** 명령어입니다. 많은 분이 `git pull`과 헷갈려하시지만, 이 둘의 차이를 이해하는 것이 Git 워크플로우를 관리하는 데 매우 중요합니다.

---

### 1. 기본적인 개념

`git fetch`를 실행하면 Git은 원격 저장소에 있는 새로운 커밋, 브랜치, 태그 정보들을 로컬로 다운로드합니다. 하지만 **현재 내가 작업 중인 코드(Working Directory)나 브랜치에 자동으로 병합(Merge)하지 않습니다.**

- **git fetch:** "저쪽 동네(Remote)에 무슨 일이 생겼는지 정보만 업데이트해줘. 내 코드는 건드리지 말고."
    
- **git pull:** "저쪽 정보를 가져온 다음(fetch), 내 코드에 바로 합쳐줘(merge)."
    

따라서 `git fetch`를 하면 원격의 내용은 로컬 저장소의 `origin/main` 같은 **원격 추적 브랜치**에만 반영됩니다.

---

### 2. 어떤 상황에서 유용한가?

#### ① 안전한 업데이트 확인 (검토 후 병합)

동료들이 올린 코드가 내 코드와 충돌(Conflict)이 날까 봐 걱정될 때 사용합니다. `fetch`로 정보를 가져온 뒤, `git diff`를 사용해 어떤 부분이 바뀌었는지 미리 확인하고 안전할 때 병합할 수 있습니다.

> `git fetch origin` → `git diff main origin/main` (차이점 확인) → `git merge origin/main`

#### ② 원격에 새로 생성된 브랜치 확인

팀원이 새로운 브랜치를 만들어 Push했을 때, 내 로컬에서는 그 브랜치가 보이지 않습니다. 이때 `fetch`를 하면 원격 브랜치 목록이 업데이트되어 해당 브랜치로 `checkout` 할 수 있게 됩니다.

#### ③ 작업 내용 유지 (Context Switching)

현재 내가 수정한 코드가 많아서 당장 `pull`을 받아 병합하기 곤란한 상황일 때 사용합니다. 우선 `fetch`만 해두면, 나중에 내 작업을 마치고 병합할 준비가 되었을 때 최신 상태를 즉시 반영할 수 있습니다.

#### ④ 오프라인 작업 대비

비행기를 타거나 인터넷이 안 되는 곳으로 가기 전에 `fetch`를 실행해두면, 인터넷이 없어도 `origin/main`과 내 `main` 브랜치를 비교하거나 과거 이력을 조회하는 등의 작업을 로컬에서 수행할 수 있습니다.

---

### 💡 요약

- **신중하고 안전하게** 관리하고 싶다면: `git fetch` 후 상황 보고 `merge`
    
- **빠르고 편리하게** 일단 합치고 싶다면: `git pull`
    

협업 규모가 커질수록 `pull`보다는 `fetch`를 통해 변경 사항을 먼저 파악하는 습관을 들이는 것이 코드 충돌 사고를 줄이는 좋은 방법입니다.

## git pull

**git pull`은 내부적으로 `git fetch`를 실행한 직후에 `git merge`를 결합하여 수행하는 명령어입니다.

두 방식의 차이점과 작동 원리를 이해하면 Git을 훨씬 더 안전하게 사용하실 수 있습니다.

---

### 1. 공식적인 관계
수식으로 표현하면 다음과 같습니다.
> **`git pull` = `git fetch` + `git merge`**


---

### 2. 단계별 차이점

#### **`git fetch` + `git merge` (수동 방식)**
1.  **`git fetch`**: 원격 저장소의 최신 이력(커밋, 브랜치 정보 등)을 로컬로 가져옵니다. 이때 내 실제 작업 파일(Working Directory)은 전혀 바뀌지 않습니다. 오직 `origin/main` 같은 **원격 추적 브랜치**만 업데이트됩니다.
2.  **검토**: `git log`나 `git diff`를 통해 동료가 어떤 코드를 수정했는지 미리 확인해 볼 수 있습니다.
3.  **`git merge`**: 확인이 끝난 후 안전하다고 판단되면 내 로컬 브랜치에 합칩니다.

#### **`git pull` (자동 방식)**
* 위 과정을 한 번에 수행합니다. 원격의 내용을 가져오자마자 현재 내가 작업 중인 브랜치에 **강제로 합치기**를 시도합니다.

---

### 3. 왜 `git pull`보다 `fetch && merge`를 추천하기도 하나요?

개발자들 사이에서 `git pull`보다 `fetch`를 먼저 하는 습관을 권장하는 이유는 **'예상치 못한 충돌(Conflict)'** 때문입니다.

* **`git pull` 사용 시**: 가져오자마자 내 코드와 합쳐버리기 때문에, 충돌이 발생하면 당황스러운 상태에서 바로 해결해야 합니다.
* **`fetch` 후 `merge` 사용 시**: `fetch`로 정보를 먼저 본 뒤, "아, 이 부분은 내가 수정한 곳과 겹치겠구나"라고 인지한 상태에서 전략적으로 `merge`를 진행할 수 있습니다.

---

### 4. 요즘의 트렌드: `git pull --rebase`
만약 커밋 이력을 한 줄로 깔끔하게 유지하고 싶다면(앞서 배우신 `rebase` 활용), `merge` 대신 `rebase`를 결합한 `pull`을 사용하기도 합니다.

* **명령어**: `git pull --rebase`
* **의미**: `git fetch` + `git rebase`

---

**요약하자면:**
`git pull`은 편리함을 위해 두 명령어를 합쳐놓은 것이 맞습니다. 하지만 프로젝트가 복잡해지거나 협업 인원이 많아질수록, **`fetch`로 먼저 간을 보고 `merge`로 식사를 하는 것**이 코드의 안정성을 지키는 데 훨씬 유리합니다.

---
---
`git fetch`를 마친 후, 내 코드와 원격의 코드가 충돌할지 미리 확인하는 방법은 매우 영리한 습관입니다. `merge`를 실행하기 전에 **"미리보기"**를 하는 몇 가지 핵심 방법을 알려드릴게요.

---

### 1. `git log`로 차이점 확인하기

가장 먼저 원격에 어떤 커밋들이 추가되었는지 목록을 확인합니다.

Bash

```
# 내 로컬 브랜치(main)에는 없고 원격(origin/main)에만 있는 커밋 보기
git log main..origin/main --oneline
```

- 이 명령어를 통해 어떤 파일들이 수정되었는지 대략적으로 파악할 수 있습니다.
    

---

### 2. `git diff`로 구체적인 코드 충돌 가능성 확인 (가장 추천)

파일의 어떤 부분이 바뀌었는지 직접 비교하는 것이 충돌 여부를 판단하는 가장 확실한 방법입니다.

Bash

```
# 내 현재 코드와 원격 브랜치의 차이점 비교
git diff main origin/main
```

**💡 팁:** 너무 내용이 많다면 **파일명만** 먼저 확인하세요.

Bash

```
# 수정된 파일 목록만 보기
git diff main origin/main --name-status
```

- 여기서 내가 현재 수정 중인 파일과 원격에서 바뀐 파일이 겹친다면, `merge` 시 충돌(Conflict)이 발생할 확률이 높습니다.
    

---

### 3. `git merge --no-commit --no-ff` (가상 머지)

실제로 합쳐보되, 기록을 남기지 않고 충돌 여부만 확인하는 트릭입니다.

Bash

```
git merge origin/main --no-commit --no-ff
```

- **충돌이 없다면:** 자동으로 스테이징 영역에 합쳐집니다. 확인 후 `git merge --abort`로 취소하면 이전 상태로 깔끔하게 돌아옵니다.
    
- **충돌이 있다면:** 터미널에 `CONFLICT (content): Merge conflict in [파일명]` 메시지가 뜹니다. 이때도 마찬가지로 `git merge --abort`를 입력하면 충돌 전으로 돌아갈 수 있습니다.
    

---

### 4. 시각화 도구 활용하기

명령어 라인이 복잡하다면 GUI의 도움을 받는 것이 직관적입니다.

- **옵시디언 Git 플러그인:** 사이드바에서 변경된 파일 목록을 시각적으로 보여줍니다.
    
- **VS Code:** 왼쪽 소스 제어 탭에서 `Fetch` 후 '상단/하단 화살표' 숫자를 클릭하면 비교 창이 뜹니다.
    

---

### 💡 정리하자면

가장 권장하는 루틴은 다음과 같습니다:

1. `git fetch origin`
    
2. `git diff main origin/main --name-only` (파일 중복 확인)
    
3. 겹치는 파일이 있다면 `git diff main origin/main [파일명]`으로 코드 확인
