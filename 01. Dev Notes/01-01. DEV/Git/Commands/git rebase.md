`git rebase`는 말 그대로 **커밋의 베이스(Base)를 다시 설정(Re)**하는 작업입니다.

`git merge`가 두 브랜치의 최종 결과물을 합쳐 '새로운 커밋(Merge Commit)'을 만드는 방식이라면, `rebase`는 내 브랜치의 시작점을 원격 브랜치의 최신 커밋으로 옮겨서 **마치 처음부터 최신 브랜치에서 작업을 시작한 것처럼** 커밋 이력을 한 줄로 깔끔하게 정리해 줍니다.

---

### 1. 기본적인 개념

브랜치 `feature`가 `main`에서 갈라져 나온 뒤, `main`에 새로운 커밋들이 추가되었다고 가정해 보겠습니다.

- **Merge:** `main`의 변경 내용을 가져와 `feature` 브랜치에 합칩니다. 이 과정에서 "Merge branch 'main' into feature"라는 커밋이 생기며 이력이 갈래갈래 나뉩니다.
    
- **Rebase:** `feature` 브랜치에서 작업한 커밋들을 잠시 떼어내고, `feature`의 뿌리(Base)를 `main`의 최신 커밋으로 바꾼 뒤 그 위에 떼어냈던 커밋들을 차례대로 다시 붙입니다.
    

결과적으로 이력이 꺾이지 않고 **직선**으로 유지됩니다.

---

### 2. 어떤 상황에서 유용한가?

#### ① 커밋 히스토리를 깔끔하게 유지하고 싶을 때 (Clean History)

여러 명이 협업할 때 모든 변경 사항을 `merge`로만 합치면 거미줄처럼 복잡한 그래프가 그려집니다. `rebase`를 사용하면 프로젝트 전체의 흐름을 파악하기 쉬운 **선형적인(Linear) 히스토리**를 만들 수 있습니다.

#### ② Pull Request(PR)를 보내기 전 최신 코드 반영

내가 작업하는 동안 `main` 브랜치에 이미 많은 변화가 생겼다면, 내 작업을 `main`의 최신 상태 위로 `rebase`하여 충돌을 미리 해결하고 PR을 보낼 수 있습니다. 이렇게 하면 코드 리뷰어가 변경 사항을 훨씬 명확하게 볼 수 있습니다.

#### ③ 로컬 커밋 정리 (Interactive Rebase)

서버에 Push하기 전, "오타 수정", "테스트" 등 자질구레하게 쌓인 로컬 커밋들을 하나로 합치거나(`squash`), 메시지를 수정하고 싶을 때 `git rebase -i`를 사용하면 매우 유용합니다.

---

### ⚠️ 주의사항 (Golden Rule)

**"이미 원격 저장소(GitHub 등)에 Push된 커밋은 절대 Rebase 하지 마세요."**

`rebase`는 기존 커밋을 삭제하고 새로운 커밋을 만드는 과정입니다. 이미 공유된 커밋을 Rebase 해버리면, 다른 팀원들의 이력과 내 이력이 완전히 꼬여버리는 대참사가 발생할 수 있습니다. 로컬에서 작업 중인 나만의 브랜치에서만 사용하는 것이 안전합니다.

---

### 💡 요약

- **Merge:** 기록 보존이 중요할 때 (합쳐진 시점을 명확히 남김)
    
- **Rebase:** 기록의 가독성이 중요할 때 (한 줄로 깔끔하게 정리)
    

소프트웨어 개발자로서 깔끔한 커밋 로그는 협업의 매너이자 실력이기도 하니, 로컬 작업 시에는 `rebase`를 적극 활용해 보시는 것을 추천합니다!
