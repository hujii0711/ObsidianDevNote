## 머지 전에 충돌여부를 체크 방법


### 1. `--no-commit` + `--no-ff` 옵션 (가장 추천)

```bash
git checkout B
git merge --no-commit --no-ff A
```

- 실제로 머지를 시도하지만 **커밋은 하지 않음**
- 충돌 여부만 확인 후 아래 명령으로 **되돌리기** 가능

```bash
git merge --abort   # 충돌 시 취소
git reset --merge   # 충돌 없을 때 취소
```

### 2. `git diff`로 사전 비교

```bash
git diff B..A
```

- 두 브랜치 간 **변경된 내용**을 미리 확인
- 같은 파일/같은 라인이 양쪽에서 다르게 수정됐다면 충돌 가능성이 높음

### 3. `git merge-tree` (실제 머지 없이 시뮬레이션)

```bash
git merge-tree $(git merge-base B A) B A
```

- 브랜치를 전혀 건드리지 않고 **충돌 여부만 시뮬레이션**
- 출력에 `<<<<<<` 마커가 보이면 충돌 있음

4. 선택 기준

| 방법 | 브랜치 변경 | 되돌리기 필요 | 추천 상황 |
|------|------------|--------------|----------|
| `--no-commit` | O | O | 충돌 내용도 보고 싶을 때 |
| `git diff` | X | X | 빠르게 차이만 볼 때 |
| `merge-tree` | X | X | 브랜치를 건드리기 싫을 때 |

가장 안전한 방법은 **`merge-tree`** 로, 현재 작업 중인 브랜치에 아무 영향을 주지 않습니다.

`git diff`만 단독으로 실행하면 브랜치 비교가 아닙니다.

---

## git diff 단독 실행 시

**현재 브랜치**에서 **워킹 디렉토리(수정했지만 아직 `git add` 안 한 파일)** 와 **스테이징 영역** 을 비교합니다.

```
워킹 디렉토리 vs 스테이징(index)
```

### 상황별 비교 대상

| 명령어                 | 비교 대상               |
| ------------------- | ------------------- |
| `git diff`          | 워킹 디렉토리 vs 스테이징     |
| `git diff --staged` | 스테이징 vs 최근 커밋(HEAD) |
| `git diff HEAD`     | 워킹 디렉토리 vs 최근 커밋    |
| `git diff B..A`     | B브랜치 vs A브랜치        |즉, **브랜치 간 비교를 하려면 반드시 브랜치명을 명시**해야 합니다.
```bash
git diff B..A   # B 기준으로 A의 변경사항 확인
```

## Git 브랜치 충돌 해결 방법

### 1. 충돌 해결 방법 (3가지 선택)

**① 직접 편집** — 가장 일반적
```bash
# 충돌 파일을 열어 마커 제거 후 원하는 코드만 남기기
# 마커(<<<<, ====, >>>>)를 모두 삭제하고 저장
```

**② VSCode 사용** — 클릭으로 간편하게
```
충돌 파일 열기 → "Accept Current / Incoming / Both Changes" 버튼 클릭
```

**③ git checkout으로 한쪽 선택**
```bash
git checkout --ours 파일명    # A브랜치(현재) 내용으로 덮어쓰기
git checkout --theirs 파일명  # B브랜치 내용으로 덮어쓰기
```

### 2. 해결 후 마무리

```bash
git add 충돌해결한파일명   # 해결한 파일 스테이징
git commit                 # 머지 커밋 완료
```

### 3. 전체 흐름 요약

```bash
git checkout A             # A브랜치로 이동
git merge B                # B 머지 시도 → 충돌 발생
# → 충돌 파일 수정
git add .
git commit                 # 완료
```

**팁:** 충돌을 줄이려면 머지 전에 `git pull`로 최신 상태를 유지하고, 같은 파일을 여러 명이 동시에 수정하는 상황을 피하는 게 좋습니다.
