
Git에서 **Tag(태그)**는 저장소의 커밋 히스토리 중 특정 지점을 **중요하다고 표시(Mark)**해두는 이정표와 같은 기능입니다.

일반적으로 소프트웨어의 **릴리스 버전(예: v1.0, v2.1)**을 관리할 때 가장 많이 사용됩니다.

---

## 1. Tag의 두 가지 종류

Git 태그는 크게 두 가지 방식으로 나뉩니다.

### ① Lightweight Tag (경량 태그)

- **특징:** 단순히 특정 커밋을 가리키는 포인터입니다. 브랜치와 비슷하지만, 브랜치처럼 커밋에 따라 앞으로 이동하지 않고 고정되어 있습니다.
    
- **용도:** 임시로 특정 지점을 표시하거나 로컬에서 짧게 관리할 때 유용합니다.
    

### ② Annotated Tag (주석 태그)

- **특징:** Git 데이터베이스에 전체 객체로 저장됩니다. 태그를 만든 사람의 이름, 이메일, 날짜, 태그 메시지를 포함하며 GPG로 서명할 수도 있습니다.
    
- **용도:** 공식적인 릴리스 버전을 배포할 때 권장되는 방식입니다.
    

---

## 2. 어떤 상황에서 유용한가?

### ① 버전 관리 및 배포 (Releases)

가장 대표적인 사용 사례입니다. 특정 시점의 코드가 테스트를 통과하고 배포 준비가 되었다면, 해당 커밋에 `v1.0.1` 같은 태그를 붙입니다.

- **유용성:** 나중에 수천 개의 커밋 사이에서 "그때 배포했던 코드가 뭐였지?"라고 찾을 필요 없이 태그 이름으로 즉시 돌아갈 수 있습니다.
    

### ② CI/CD 파이프라인 자동화

많은 배포 자동화 도구들이 **태그가 생성되는 것을 감지(Trigger)**하여 자동으로 빌드와 배포를 수행합니다.

- **상황:** `v`로 시작하는 태그가 푸시되면 운영 서버에 배포하고, `beta` 태그가 푸시되면 테스트 서버에 배포하도록 설정할 수 있습니다.
    

### ③ 특정 시점의 스냅샷 저장

대규모 리팩토링을 시작하기 전이나, 중요한 이정표가 되는 기능을 완성했을 때 기록용으로 남깁니다.

- **유용성:** 브랜치는 계속 변하지만, 태그는 한 번 붙이면 삭제하지 않는 한 그 자리에 고정되어 있어 신뢰할 수 있는 스냅샷 역할을 합니다.

## 3. 핵심 명령어 요약

- **태그 목록 보기:** `git tag`
    
- **주석 태그 생성:** `git tag -a v1.0.0 -m "첫 번째 정식 버전 배포"`
    
- **태그 정보 확인:** `git show v1.0.0`
    
- **태그 원격에 푸시:** `git push origin v1.0.0` (기본 `push`로는 태그가 전송되지 않습니다.)
    
- **모든 태그 푸시:** `git push origin --tags`
    

---

## 4. Tag vs Branch: 무엇이 다른가?

|**구분**|**Branch (브랜치)**|**Tag (태그)**|
|---|---|---|
|**성격**|**살아있는 선**: 새로운 커밋이 추가되면 함께 이동|**박제된 점**: 한 번 붙으면 이동하지 않음|
|**목적**|기능 개발, 버그 수정 등 진행 중인 작업|특정 시점의 기록, 릴리스 버전 표시|
|**이동**|`checkout` 후 커밋하면 자동으로 포인터 이동|원칙적으로 이동하지 않으며, 필요시 삭제 후 다시 생성|

## 💡 팁: Semantic Versioning (SemVer)

태그 이름을 정할 때는 `v1.2.3`과 같은 형식을 많이 사용합니다.

- **1 (Major):** 하위 호환성이 깨지는 큰 변화
    
- **2 (Minor):** 기능 추가 (하위 호환성 유지)
    
- **3 (Patch):** 단순 버그 수정
    

혹시 현재 진행 중인 프로젝트에서 버전 관리 규칙을 정해두고 사용하고 계신가요? 한 번 설정해두면 나중에 `gh release create` 같은 도구와 연동하여 배포 노트를 자동으로 생성할 때도 매우 편리합니다.
