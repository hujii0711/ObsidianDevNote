
Git 저장소에서 **Issue(이슈)**는 프로젝트와 관련된 모든 작업 단위, 버그, 개선 사항, 그리고 아이디어를 기록하고 관리하는 **협업 및 프로젝트 관리 도구**입니다.

엄밀히 말하면 Git 자체의 기능이라기보다는 GitHub, GitLab, Bitbucket 같은 Git 호스팅 플랫폼에서 제공하는 기능이지만, 현대 소프트웨어 개발 흐름에서는 Git 워크플로우의 필수적인 부분으로 자리 잡고 있습니다.

---

## 1. Issue의 핵심 구성 요소

이슈는 단순한 게시판 글 이상의 구조화된 정보를 담고 있습니다.

- **Label(라벨):** `bug`, `enhancement`, `documentation`, `question` 등 이슈의 성격을 분류합니다.
    
- **Assignee(담당자):** 해당 작업을 책임지고 처리할 사람을 지정합니다.
    
- **Milestone(마일스톤):** 특정 날짜나 버전(예: v1.0 출시)까지 해결해야 할 이슈들을 그룹화하여 진행률을 관리합니다.
    
- **Reference(참조):** 이슈 번호(예: `#123`)를 통해 다른 이슈나 Pull Request, 특정 커밋과 서로 연결할 수 있습니다.
    

---

## 2. 어떤 상황에서 유용한가?

### ① 버그 추적 (Bug Tracking)

프로젝트 운영 중 발견된 문제점을 기록합니다. 누가 발견했는지, 어떤 환경에서 발생하는지, 해결되었는지 등을 한눈에 파악할 수 있습니다.

- **예시:** "로그인 버튼 클릭 시 500 에러 발생"이라는 이슈를 생성하고, 해결 후 관련 코드를 커밋할 때 이슈 번호를 남깁니다.
    

### ② 새로운 기능 제안 (Feature Request)

다음에 추가할 기능이나 개선 아이디어를 기록하고 팀원들과 논의하는 장으로 활용합니다.

- **예시:** "다크 모드 지원 추가", "검색 결과 필터링 기능 개선" 등.
    

### ③ 작업 할당 및 칸반 보드 연동

이슈를 생성하고 담당자를 지정함으로써 프로젝트 내에서 **누가 어떤 작업을 하고 있는지** 명확히 합니다. 대부분의 플랫폼은 이슈를 카드 형태로 시각화하는 'Project Board(칸반 보드)' 기능을 제공하여 전체 워크플로우를 관리하게 해줍니다.

### ④ 문서화 및 히스토리 관리

과거에 특정 버그를 어떻게 해결했는지, 왜 이 기능이 도입되었는지에 대한 **토론 기록**이 남습니다. 이는 나중에 프로젝트에 새로 합류한 개발자에게 훌륭한 참고 자료가 됩니다.

---

## 3. Git 커밋과 Issue의 강력한 연결

이슈의 진정한 가치는 Git 커밋 메시지와의 연동에서 나옵니다.

- **자동 종료:** 커밋 메시지에 `Fixes #10` 또는 `Closes #10`이라고 적고 원격 저장소에 푸시하면, 10번 이슈가 자동으로 **Closed** 상태로 변경됩니다.
    
- **추적성:** 특정 코드 줄이 왜 수정되었는지 궁금할 때, 해당 커밋과 연결된 이슈를 찾아가면 당시의 논의 배경과 요구사항을 즉시 확인할 수 있습니다.

## 4. 활용 팁: Issue Template

프로젝트 설정에서 **Issue Template**을 만들면, 사용자가 이슈를 작성할 때 "버그 재현 경로", "기대 결과", "스크린샷" 등 정해진 양식에 맞춰 내용을 채우도록 유도할 수 있습니다. 이는 불필요한 질문과 답변 과정을 줄여 협업 효율을 극대화합니다.

혹시 현재 진행 중인 프로젝트에서 이슈를 사용해 팀원들과 업무를 분담하거나 로드맵을 관리해본 경험이 있으신가요?

---
Git 이슈(Issue)를 등록하는 방법은 크게 **웹 인터페이스(GitHub/GitLab UI)**를 사용하는 방법과 **명령행 도구(CLI)**를 사용하는 방법으로 나뉩니다.

---

## 1. GitHub 웹 사이트에서 등록하기 (가장 일반적인 방법)

브라우저를 통해 시각적으로 확인하며 등록하는 방법입니다.

1. **저장소(Repository) 접속:** 이슈를 등록할 프로젝트 저장소로 이동합니다.
    
2. **Issues 탭 클릭:** 상단 메뉴 바에서 **[Issues]** 탭을 클릭합니다.
    
3. **New Issue 클릭:** 오른쪽 상단의 초록색 **[New issue]** 버튼을 누릅니다.
    
    - 만약 프로젝트에 **Issue Template**이 설정되어 있다면, 보고하려는 유형(Bug, Feature 등)을 먼저 선택하게 됩니다.
        
4. **내용 작성:**
    
    - **Title:** 이슈의 핵심 내용을 간결하게 적습니다. (예: `[Bug] 로그인 페이지 모바일 레이아웃 깨짐`)
        
    - **Write:** 상세 내용을 적습니다. 마크다운(Markdown) 형식을 지원하므로 이미지나 코드 블록을 넣을 수 있습니다.
        
5. **사이드바 설정 (선택 사항):**
    
    - **Assignees:** 누구에게 이 작업을 맡길지 지정합니다.
        
    - **Labels:** 이슈의 성격(bug, enhancement 등)을 분류합니다.
        
    - **Projects/Milestones:** 진행 상황이나 특정 출시 일정에 연결합니다.
        
6. **제출:** 하단의 **[Submit new issue]**를 누르면 등록이 완료됩니다.

## 2. GitHub CLI(`gh`)를 통해 등록하기

터미널에서 코딩 흐름을 끊지 않고 바로 등록할 수 있어 개발자들에게 매우 효율적입니다.

### 기본 명령어

터미널에서 아래 명령어를 입력합니다.

Bash

```
gh issue create --title "이슈 제목" --body "상세 내용"
```

### 대화형 모드 (추천)

옵션 없이 명령어만 입력하면 제목과 내용을 터미널 내에서 대화형으로 입력할 수 있습니다.

Bash

```
gh issue create
```

- 명령어 입력 후 **Title**, **Body**를 차례로 입력합니다.
    
- 마지막에 `Submit`, `Continue in browser`, `Add metadata`(라벨, 담당자 등 지정) 중 하나를 선택할 수 있습니다.

## 3. 이슈 등록 시 꿀팁

### ① 이슈 템플릿(Issue Templates) 활용

협업 시 팀원들이 일관된 형식으로 이슈를 올리게 하려면 저장소 루트의 `.github/ISSUE_TEMPLATE/` 폴더에 템플릿 파일(`bug_report.md` 등)을 만들어 두는 것이 좋습니다.

### ② 체크리스트 사용

이슈 본문에 아래와 같이 작성하면 할 일 목록(Checkbox)이 생성되어 진행 상황을 체크하기 좋습니다.

Markdown

```
- [x] UI 디자인 확인
- [ ] API 연동 완료
- [ ] 단위 테스트 수행
```

### ③ 키워드를 통한 자동 연결

이슈 본문에 다른 이슈 번호를 언급하면(예: `#42`) 자동으로 링크가 생성됩니다. 나중에 이 이슈를 해결하는 Pull Request를 보낼 때 본문에 `Closes #42`라고 적으면, PR이 머지될 때 해당 이슈가 자동으로 닫힙니다.

---

**참고:** 만약 회사 내에서 **GitLab**이나 **Bitbucket**을 사용 중이라면 인터페이스의 위치는 조금씩 다르지만, [Issues] 메뉴 -> [New Issue] 버튼이라는 기본 흐름은 거의 동일합니다.
