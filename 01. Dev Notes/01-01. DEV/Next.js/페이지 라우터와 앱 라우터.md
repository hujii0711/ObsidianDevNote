
### 1. 디렉토리 구조

**페이지 라우터 (Pages Router)**는 `pages/` 디렉토리를 사용합니다. <font color="#ff0000">파일 하나가 곧 라우트 하나</font>입니다.

```
pages/
├── index.tsx          → /
├── about.tsx          → /about
├── blog/
│   ├── index.tsx      → /blog
│   └── [slug].tsx     → /blog/:slug
└── api/
    └── hello.ts       → /api/hello
```

**앱 라우터 (App Router)**는 `app/` 디렉토리를 사용하며, <font color="#ff0000">폴더가 라우트</font>이고 그 안의 특수 파일들이 역할을 분담합니다.

```
app/
├── page.tsx           → /
├── layout.tsx         → 루트 레이아웃
├── about/
│   └── page.tsx       → /about
└── blog/
    ├── layout.tsx     → /blog 전용 레이아웃
    └── [slug]/
        └── page.tsx   → /blog/:slug
```

### 2. 렌더링 방식

|구분|페이지 라우터|앱 라우터|
|---|---|---|
|기본값|CSR (클라이언트)|**RSC** (서버 컴포넌트)|
|서버 렌더링|`getServerSideProps`|`async` 컴포넌트|
|정적 생성|`getStaticProps`|기본 동작 (fetch 캐싱)|
|클라이언트|기본값|`"use client"` 명시 필요|

**페이지 라우터** — 데이터 패칭 함수가 컴포넌트 밖에 분리됩니다.

```tsx
// pages/blog/[slug].tsx
export async function getServerSideProps({ params }) {
  const post = await fetchPost(params.slug);
  return { props: { post } };
}

export default function BlogPost({ post }) {
  return <article>{post.title}</article>;
}
```

**앱 라우터** — 컴포넌트 자체가 `async`이고, 직접 데이터를 가져옵니다.

```tsx
// app/blog/[slug]/page.tsx
export default async function BlogPost({ params }) {
  const post = await fetchPost(params.slug); // 서버에서 직접 실행
  return <article>{post.title}</article>;
}
```

### 3. 레이아웃 시스템

**페이지 라우터**는 `_app.tsx`로 전역 레이아웃을 관리하고, 중첩 레이아웃은 수동으로 구현해야 합니다.

```tsx
// pages/_app.tsx — 모든 페이지에 적용되는 단일 레이아웃
export default function App({ Component, pageProps }) {
  return (
    <GlobalLayout>
      <Component {...pageProps} />
    </GlobalLayout>
  );
}
```

**앱 라우터**는 각 폴더에 `layout.tsx`를 두어 중첩 레이아웃을 자연스럽게 표현합니다. 리렌더링 없이 레이아웃이 유지됩니다.

```tsx
// app/dashboard/layout.tsx — /dashboard/* 에만 적용
export default function DashboardLayout({ children }) {
  return (
    <div>
      <Sidebar />      {/* 페이지 이동해도 유지됨 */}
      <main>{children}</main>
    </div>
  );
}
```

### 4. 특수 파일 (앱 라우터 전용)

앱 라우터는 역할별로 파일을 분리합니다.

| 파일명             | 역할                   |
| --------------- | -------------------- |
| `page.tsx`      | 라우트의 UI (공개 접근 가능)   |
| `layout.tsx`    | 중첩 레이아웃 (상태 유지)      |
| `loading.tsx`   | Suspense 로딩 UI       |
| `error.tsx`     | Error Boundary       |
| `not-found.tsx` | 404 UI               |
| `template.tsx`  | 매 이동마다 새로 마운트되는 레이아웃 |

```tsx
// app/blog/loading.tsx — 자동으로 Suspense 처리
export default function Loading() {
  return <Skeleton />;
}

// app/blog/error.tsx — 자동으로 Error Boundary 처리
"use client"
export default function Error({ error, reset }) {
  return <button onClick={reset}>다시 시도</button>;
}
```

### 5. 데이터 패칭 & 캐싱

**페이지 라우터**는 페이지 단위로만 캐싱 전략을 설정할 수 있습니다.

```tsx
// ISR — 60초마다 재생성
export async function getStaticProps() {
  return { props: { data }, revalidate: 60 };
}
```

**앱 라우터**는 `fetch` 자체에 캐싱 옵션을 부여해 컴포넌트 단위로 세밀하게 제어합니다.

```tsx
// 각 fetch마다 다른 전략 적용 가능
const staticData  = await fetch(url, { cache: 'force-cache' });       // SSG
const dynamicData = await fetch(url, { cache: 'no-store' });          // SSR
const isrData     = await fetch(url, { next: { revalidate: 60 } });   // ISR
```

### 6. Server Actions (앱 라우터 전용)

API 라우트 없이 서버 함수를 직접 호출할 수 있습니다.

```tsx
// app/actions.ts
"use server"
export async function createPost(formData: FormData) {
  await db.post.create({ data: { title: formData.get('title') } });
  revalidatePath('/blog');
}

// app/blog/new/page.tsx
import { createPost } from '../actions';

export default function NewPost() {
  return (
    <form action={createPost}>
      <input name="title" />
      <button type="submit">작성</button>
    </form>
  );
}
```

### 7. 언제 무엇을 선택할까?

|상황|권장|
|---|---|
|신규 프로젝트|✅ 앱 라우터|
|기존 프로젝트 유지보수|페이지 라우터 유지|
|복잡한 중첩 레이아웃|✅ 앱 라우터|
|서버 중심 데이터 패칭|✅ 앱 라우터|
|팀이 RSC에 익숙하지 않음|페이지 라우터로 시작 후 마이그레이션|

> **핵심 요약:** 앱 라우터는 **React Server Components**를 기반으로 서버/클라이언트 경계를 명확히 하고, 레이아웃·로딩·에러를 파일 단위로 선언적으로 관리하는 현대적인 방식입니다. Next.js 13+에서 도입되어 현재 공식 권장 방식입니다.
