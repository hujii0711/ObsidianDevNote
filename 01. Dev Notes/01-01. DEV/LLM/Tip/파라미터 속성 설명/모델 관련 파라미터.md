PreTrainedConfig 클래스

PreTrainedConfig는 허깅페이스의 사전 학습된 모델들의 구성을 정의하는 기본 클래스이다.
허깅페이스의 사전 학습된 모델들에서 공통으로 사용되는 기본 구성 클래스로, 모델의 구조와 하이퍼파라미터를 간편하게 정의하고 관리할 수 있다.

1. model_type: 모델의 유형을 나타내는 문자열이다.(ex. bert, gpt2등)

2. vocab_size: 모델의 어위 사전 크기이다.
모델이 인식할 수 있는 고유 토큰의 수를 결정한다.

3. hidden_size: 모델의 은닉 계층에 있는 노드의 수이다.
이 값이 클수록 모델의 표현 능력이 높아진다.

4. num_attension_heads: 모델의 멀티 헤드 어텐션에서 사용되는 어텐션 헤드의 수이다.

5. num_hidden_layers: 모델의 트랜스포머 계층 수이다.
계층의 수가 많을수록 모델의 표현 능력이 높아진다.

6. output_hidden_states: 모델이 모든 은닉 상태를 출력할지를 결정한다.

7. output_attensions: 모델이 모든 어텐션 값을 출력할지를 결정한다.

8. return_dict: 모델이 일반 튜플 대신 ModelOutput 객체를 반환할지를 결정한다.

9. is_encoder_decoder: 모델이 인코더-디코더 모델인지를 나타낸다.

10. is_decoder: 모델이 디코더 모델인지를 나타낸다.

PreTrainedTokenizer 클래스

1. vocab_files_names: 단어 집합 파일의 이름과 경로를 포함하는 딕셔너리이다.
ex) {"vocab_file": "path/to/vocab.txt"}과 같은 형식으로 경로를 지정한다.

2. pretrained_vocab_files_map: 사전 학습된 단어 집합 파일의 매핑을 포함하는 딕셔너리이다.
키는 모델의 이름이나 버전을, 값은 각 파일의 이름과 경로를 포함한다.
ex) {"bert-base-uncased": {"vocab_file": "path/to/vocab.txt"}}과 같은 형식으로 경로를 지정한다.

3. pretrained_init_configuration: 사전 학습된 토크나이저 구성을 포함하는 딕셔너리이다.
키는 모델의 이름이나 버전을, 값은 해당 모델의 토크나이저 구성을 나타낸다.

4. max_model_input_sizes: 모델의 최대 입력 길이를 지정하는 딕셔너리이다.
키는 모델의 이름이나 버전을, 값은 해당 모델의 최대 입력 길이를 나타내는 정숫값을 입력한다.
만약 None 값이라면 입력 길이에 제한이 없음을 의미한다.

5. model_max_length: 토크나이저가 사용하는 모델의 최대 입력 길이를 지정한다.
이 값을 기준으로 입력 시퀀스를 자르거나 패딩한다.

6. padding_side: 입력 시퀀스에 패딩을 적용할 위치를 지정한다.
"left"인 경우 왼쪽에 패딩을 추가하고 "right"인 경우 오른쪽에 패딩을 추가한다.

7. truncation_side: 입력 시퀀스가 model_max_length를 초과할 때, 어느 쪽에서 자를지 결정한다.
"left"인 경우 왼쪽에서 자르고, "right"인 경우 오른쪽에서 자른다.

8. model_input_names: 순전파에 입력되는 텐서들의 이름 목록을 설정한다.
예를 들어, BERT 모델의 경우 ["input_ids", "attension_mask", "token_type_ids"]와 같이 지정된다.

9. bos_token: 시퀀스의 시작을 나타내는 BOS(Beginning Of Sequence) 토큰을 설정한다.

10. eos_token: 시퀀스의 끝을 나타내는 EOS 토큰을 설정한다.

11. unk_token: 단어 집합에 없는 토큰을 대체하는 UNK(Unknowm) 토큰을 설정한다.

12. sep_token: 두 개의 시퀀스를 구분하는 SEP 토큰을 설정한다.

13. pad_token: 시퀀스를 패딩할 때 사용하는 PAD 토큰을 설정한다.

14. cls_token: 시퀀스 전체를 분류하는 CLS(Classification) 토큰을 설정한다.

15. mask_token: 마스크된 언어 모델링 작업에서 마스킹된 토큰을 나타내는 Mask 토큰을 설정한다.

16. additional_special_tokens: 위에 나열된 특수 토큰 외에 추가로 필요한 특수 토큰 목록을 설정한다.

Trainer 클래스
Trainer 클래스는 모델 학습 프로세스의 모든 측명을 캡슐화해 통합된 인터페이스를 제공한다.
--> 학습 루프, 평가, 하이퍼파라미터 튜닝 등의 작업을 쉽고 효율적으로 수행할 수 있다.
결국 Trainer 클래스는 실제 모델 학습 및 평가 프로세스를 관리하기 위한 클래스다. 모델, 데이터세트, 토크나이저, 최적화 단계 수행 등 학습을 위한 핵심 로직을 구현한다.
학습률, 배치 크기, 에폭수, 평가 전략 등 광범위한 학습 관련 설정은 트레이닝 아규먼트 클래스로 설정한다.


trainer = transformers.Trainer()


1. model: 학습, 평가 또는 예측에 사용할 모델 인스턴스이다.
PreTrainedModel 또는 Torch.nn.Module의 인스턴스여야 하며, 모델을 제공하지 않는 경우 model_init을 전달해야 한다.

2. args: 학습 관련 설정을 조정하기 위한 트레이닝 아규먼트 인스턴스다.
입력하지 않으면 기본 트레이닝 아큐먼트 인스턴스가 사용되며, 출력 디렉터리는 현재 디렉터리에 tmp_trainer라는 이름의 디렉터리로 설정된다.

3. data_collector: 학습 또는 평가 데이터세트에서 추출된 배치를 모델에 입력하기 전에 패딩, 마스킹 등의 작업을 수행하기 위한 함수를 입력한다.

4. trainer_dataset: 학습에 사용할 데이터세트이다.
모델의 순방향 메서드에서 허용하지 않는 열은 자동으로 제거된다. 분산 환경에서 학습할 경우, 데이터세트는 모든 프로세스에서 동일한 시드를 사용하거나 set_epoch 메서드를 구현해야 한다.

5. eval_dataset: 평가에 사용할 데이터세트이다.
딕셔너리인 경우 각 데이터세트에 대해 키를 지표 이름 앞에 추가해 평가한다.

6. tokenizer: 데이터 전처리에 사용할 PreTrainedTokenizer 인스턴스이다.
배치 생성 시 자동으로 입력을 최대 길이로 패딩하고 한다. 토크나이저는 모델과 함께 저장되어 학습이 중단돼도 재사용할 수 있다.

7. model_init: 모델 인스턴스를 생성하는 함수를 설정한다.
train 메서드가 호출될 때마다 새로운 인스턴스가 생성된다.

8. compute_metrics: 평가 시 지표를 계산하는 함수를 입력한다. 정확도나 F1 점수 등을 이용해 모델을 평가한다.

9. callbacks: 학습 프로세스 중에 특정 이벤트가 발생했을 때 실행될 사용자 정의 함수를 등록 한다.
에폭이 끝날 때마다 모델 성능을 평가하고 싶을 때나 학습 중 특정 조건에 따라 조기 종료하고 싶을 때 사용한다.

10. optimizers: 모델을 학습하는 데 사용되는 최적화 함수를 의미한다.
아무것도 입력하지 않으면 AdamW 최적화 함수를 사용한다.

11. preprocess_logins_for_metrics: 모델의 로짓 출력을 평가 지표 계산에 사용되기 전에 전처리하는 함수를 지정한다.

TrainerArguments 클래스
트레이닝 아큐먼트는 학습 하이퍼파라미터와 설정을 캡슐화한 클래스이다. 트레이너 클래스에 전달되어 학습 프로세스를 제어하는 데 사용된다.


training_args = transformers.TrainingArguments()


1. output_dir: 모델 예측 결과와 체크 포인트가 저장될 출력 디렉터리 경로를 설정한다.
cp) 학습중 만들어지는 모델 체크포인트를 저장할 폴더를 지정한다. 학습이 중간에 종료되거나 나중에 이어서 학습할 때. 이 디렉터리에 저장된 체크포인트를 불러와
작업을 재개할 수 있다.

2. overwrite_output_dir: 출력 디렉터리의 내용을 덮어쓸지 여부를 결정한다.
True로 설정하면 output_dir에 지정된 디렉터리의 기존 내용을 덮어쓴다. 이전에 학습을 중단했던 체크포인트 디렉터리에서 학습을 재개하고 싶은 경우에 주로 사용된다.

3. do_train: 모델 학습 수행 여부를 설정한다.
모델을 학습하는 경우 True로 설정한다. 

4. do_eval: 모델 학습 중 검증 데이터세트에 대한 평가 수행 여부를 설정한다.
eval_strategy가 no가 아닌 경우 True로 설정된다.

5. do_predict: 테스트 데이터세트에 대한 예측을 수행할지 여부를 지정한다.
일반적으로 모델 평가에서 사용된다.

6. eval_strategy: 모델 학습 중 평가 전략을 설정한다.
no로 설정하는 경우, 평가를 수행하지 않으며, steps로 설정한 경우 eval_steps 간격마다 평가를 수행하고 로그를 기록한다. epoch으로 설정하는 경우, 각 에폭 종료 후 평가를 수행한다.

7. per_device_train_batch_size: 모델 학습에 사용하는 장치(CPU.GPU,TPU)당 배치 크기를 의미한다.
예를 들어 2개의 GPU가 있을 때, per_device_train_batch_size를 16으로 설정하면 전체 배치 크기는 32가 된다.
cp) 학습할 때 각 GPU에서 한 번에 처리할 데이터 샘플의 수를 정한다.
배치 크기가 크면 한 번에 더 많은 데이터를 처리하지만, 그만큼 메모리 사용량도 커진다. 이는 학습 속도와 메모리 사용량 간의 균형을 맞추는 데 중요한 역할을 한다.

8. per_device_eval_batch_size: 모델 평가 시 장치당 배치 크기를 의미한다.
per_device_train_batch_size와 동일한 방식으로 작동한다.
cp) 평가할 때 GPU에서 한 번에 처리할 데이터 샘플의 수를 정한다. 이는 모델의 평가 성능을 측정할 때 사용되며, 학습과 평가를 위한 배치 크기를 각각 설정할 수 있다.

9. eval_delay: 첫 평가를 수행하기 전에 기다려야 하는 에폭 또는 스텝 수를 설정한다.

10. learning_rate: AdamW 최적화 함수의 초기 학습률을 의미한다.

11. weight_decay: AdamW 최적화 함수의 가중치 감쇠 값을 의미한다.
cp) 과적합을 막기 위해 모델 가중치를 조절하는 강도이다.
모델이 학습할 때 가중치가 너무 커지지 않도록 규제하는 역할을 하며, 적절한 값을 설정하면 모델의 일반화 성능을 높일 수 있다.

12. adam_beta1, adam_beta2, adam_epsilon: AdamW 최적화 함수의 베타1, 베타2, 엡실론 값을 의미한다.

13. max_grad_norm: 그레이디언트 클리핑을 위한 최대 그레이디언트 노름 값을 의미한다.

14. num_train_epochs: 수행할 총 에폭 수를 설정한다.

15. max_steps: 최대 학습 스텝 수이다.
양의 정수로 지정하면 num_train_epochs를 무시하고 해당 스텝까지만 학습한다.
cp) 몇 스텝을 학습할지 설정하는 옵션이다.
이 옵션은 모델이 학습하는 총 스텝 수를 제어한다.

16. warmup_steps: 웜업 단계의 스텝 수를 지정한다.
학습 초기에 학습률을 점진적으로 증가시키는 기법으로 웜업 단계 동안 학습률은 일정 스케줄(선형, 코사인)에 따라 점진적으로 증가한다.

17. logging_dir: 텐서보드 로그를 저장할 디렉터리 경로를 설정한다.
cp) 로깅 도구가 사용할 로그 파일을 어디에 저장할지 정한다. 학습 중 발생하는 로그 정보를 저장하는 디렉터리를 지정하며, 이를 통해 학습 과정을 모니터링할 수 있다.

18. logging_strategy: 모델 학습에 사용할 로깅 전략으로 no, epoch, steps를 지정할 수 있다.
cp) 100스텝마다 학습이 어떻게 진행되고 있는지 기록한다. 일정한 주기마다 학습 상태, 손실값 등을 기록해 학습 과정을 추적할 수 있게 도와준다.

19. logging_steps: logging_strategy가 steps일 때, 로그을 출력할 업데이트 스텝 간격을 의미한다.
cp) 지정한 크기 스텝마다 학습이 어떻게 진행되고 있는지 기록한다. 일정한 주기마다 학습 상태, 손실값 등을 기록해 학습 과정을 추적할 수 있게 도와준다.

20. save_strategy: 체크포인트 저장 전략으로 no, epoch, steps를 지정할 수 있다.

21. save_steps: save_strategy가 steps일 때, 체크포인트를 저장할 업데이트 스텝 간격을 의미한다.
0과 1 사이의 실수를 지정하면 총 학습 스텝의 비율로 간주된다.

22. save_total_limit: 최대 체크포인트 수이다.
지정된 수를 초과하면 가장 오래된 체크포인트를 삭제한다. 만약 load_best_model_at_end가 True이면 최적 체크포인트는 항상 보존된다.

23. seed: 학습 시작 시 설정될 무작위 시드를 의미한다.
재현성을 위해서 model_init 함수로 모델을 생성해야 한다.

24. fp16: 16비트 혼합 정밀도 학습 수행 여부를 설정한다.
True로 설정하면 32비트 대신 16비트 혼합 정밀도로 학습을 수행한다.

25. fp16_opt_level: ft16 학습시 Apex AMP(자동 혼합 정밀도) 최적화 수준을 설정한다.
01, 02 등을 지정할 수 있다.

26. run_name: WandB, MLflow 등의 로깅 도구에서 사용할 레이블을 설정한다.
설정하지 않으면 output_dir과 동일하게 할당된다.

27. load_best_model_at_end: 학습 종료 후 가장 성능이 좋았던 모델을 불러올지 여부이다.
True로 설정하면 최적 체크포인트가 항상 보존된다.

28. metric_for_best_model: 평가 지표에서 큰 값이 더 좋은 모델인지 여부를 지정한다.
예를 들어 metric_for_best_model="eval_accuracy"로 지정했다면 greater_is_better="evel_loss"라면 greater_is_better=False여야 손실이 작은 모델을 최적 모델로 인식한다.

29. optim: 모델 학습에 사용할 최적화 함수를 지정한다.
adaw_torch나 adafactor 등을 선택할 수 있다.

30. optim_args: 최적화 함수에 전달할 추가 인수를 할당한다.

31. report_to: 실험 결과와 로그를 전달할 플랫폼을 설정한다.
wandb, tensorboard, mlflow 등을 지정할 수 있다.
cp) 어떤 도구를 사용해 학습 현황을 관찰할지 정하는 곳이다. 이를 통해 학습 과정을 모니터링 할 수 있다.

32. push_to_hub: 모델이 저장될 때마다 허깅페이스 허브에 모델을 업로드할지 설정한다.

33. resume_from_checkpoint: 유효한 체크포인트가 포함된 폴더 경로를 지정한다.
resume_from_checkpoint를 지정하면 해당 경로의 체크포인트를 불러와 중단된 지점부터 학습을 재개할 수 있다.

이런 방식은 메모리 효율을 높이고, 학습 속도를 향상시키며, 과적합 위험을 줄이고, 모델의 핵심 기능을 효과적으로 조정하면서도

4-3. v_proj: 각 단어의 실제 정보를 나타낸다.

CAUSAL_LM은 인과적 모델 언어를 의미한다. 인과적이라는 표현은 모델이 오직 이전에 등장한 단어들만 고려해 예측한다는 특성을 나타낸다.
이 설정을 사용하면 지금까지 본 정보만을 바탕으로 다음에 올 내용을 예측하라고 모델에 지시하는 것과 같다.

pipeline 함수

1. task: 수행하려는 과제를 설정한다.

2. model: 파이프라인에서 사용할 모델을 설정한다.
모델의 이름이나 경로 또는 직접 PreTrainedModel(파이토치) 또는 TFPreTrainedModel(텐서플로) 객체를 전달할 수 있다.
지정하지 않으면 task에 대한 기본 모델을 사용한다.

3. config: 모델 구성 파일의 이름이나 경로 또는 PreTrainedConfig 객체를 지정한다.
지정하지 않으면 task에 대한 기본 모델 설정을 사용한다.

4. tokenizer: 파이프라인에서 사용할 토크나이저를 지정한다.
토크나이저의 이름이나 경로 또는 PreTrainedTokenizer나 PreTrainedTokenizerFast 객체를 전달한다.
지정하지 않으면 task에 대한 기본 토크나이저를 사용한다.

5. feature_extractor: 파이프라인에서 특징 추출기를 지정한다.
특징 추출기의 이름이나 경로 또는 PreTrainedFeatrueExtractor 객체를 전달한다.
지정하지 않으면 task에 대한 기본 특징 추출기를 사용한다.

6. image_processor: 파이프라인에서 사용할 이미지 프로세서를 지정한다.
이미지 프로세서의 이름이나 경로 또는 BaseImageProcessor 객체를 전달한다.

7. framework: 파이프라인에서 사용할 딥러닝 프레임워크를 지정한다.
pt, tf를 입력할 수 있다. 지정하는 않으면 설치된 프레임워크나 모델에 따라 자동으로 결정된다.

8. revision: 모델 허브에서 모델을 다운로드할 때 사용할 특정 버전을 지정한다.
브랜치 이름, 태그 이름, 커밋 ID를 입력할 수 있다.

9. use_fast: 빠른 토크나이저(PreTrainedTokenizerFast)를 사용할지 여부를 지정한다.

10. token: 모델 허브에서 모델을 다운로드할 때 사용할 액세스 토큰을 지정한다.

11. device: 파이프라인에 할당될 장치를 설정한다.
cpu, cuda:1, mps, GPU 번호 및 파이토피 장치 인스턴스를 입력할 수 있다.

12. device_map: 모델의 각 모듈을 어떤 장치에 불러올지 지정한다.
auto로 설정하면 가장 최적화된 device_map이 자동으로 계산된다. device와 device_map을 동시에 사용하면 충돌이 발생할 수 있으므로 사용에 주의한다.

13. torch_dtype: 모델의 가중치를 불러올 때 사용할 정밀도(torch.float16, torch.bfloat16 등)을 지정한다.

14. trust_remote_code: 사용자 정의 모델링, 구성, 토크나이저 또는 파이프라인 파일의 코드를 신뢰할지 여부를 지정한다.
코드를 읽고 신뢰할 수 있는 저장소에서만 True로 설정해야 한다.

15. model_kwargs: 모델을 생성할 때 추가로 전달할 매개변수를 딕셔너리 형태로 지정한다.

16. pipeline_class: 사용자 정의 파이프라인 클래스를 지정한다.

load_dataset 함수

1. path: 데이터세트의 경로 또는 이름을 설정한다.
이 매개변수의 값에 따라 load_dataset 함수가 사용하는 데이터세트 빌더가 결정된다.
- 로컬 데이터세트인 경우: 해당 디렉터리의 경로를 지정한다.
- path가 로컬 데이터세트 스크립트 파일이거나 동일한 이름의 디렉터리인 경우: 스크립트에서 데이터를 불러온다.
- 허깅페이스 허브에서 호스팅되는 공개 데이터세트인 경우: 데이터세트의 이름이나 저장소 경로를 지정한다.

2. name: 데이터세트 구성의 이름을 설정한다.
공개 데이터세트에 여러 종류가 있다면 특정 구성을 선택해 불러올 수 있다. 예를 들어, glue 데이터세트에는 cola, sst2 등의 여러 구성들이 있으며, name 인수를 통해 원하는 구성을 지정할 수 있다.

3. data_dir: 데이터세트의 디렉터리를 설정한다.
예를 들어, path의 값이 text라면 data_dir에 위치한 파일을 텍스트 형식으로 불러온다.

4. data_files: 데이터세트의 파일 경로를 지정한다.
앞선 data_dir는 디렉터리를 지정하고, data_files는 실제 데이터파일의 경로를 지정한다. 이를 통해 특정 경로의 특정 파일만 불러올 수 있다.

5. split: 불러오려는 데이터세트의 분할을 지정한다.
train, test 등을 할당할 수 있으며, train+test나 train[:10%]+train[-80%:]와 같이 여러 분할을 결합해 지정할 수도 있다.

6. cache_dir: 데이터세트 캐시를 저장할 디렉터리 경로다.
기본값은 ~/.cache/huggingface/datasets로 해당 경로에 저장된다.

7. revision: 데이터세트의 버전을 설정한다.
공개 저장소는 자체 깃허브 저장소를 가지므로 태그 이름, 브랜치 이름 또는 커밋 해시를 할당해 데이터세트의 버전을 설정할 수 있다.

8. token: 데이터세트 다운로드에 사용할 토큰을 지정한다.
특정 데이터세트에 접근하려면 해당 데이터세트 저장소의 접근 권한이 필요하다. 이 경우, 동의 버튼을 눌러야 하는데, 이를 통해 사용자 정보(이메일과 사용자 이름)를 데이터세트 저장소의 저자와 공유하게 된다.
권한이 승인되면, 데이터세트를 다운로드할 때 토큰을 사용해 인증한다.

9. streaming: 데이터 파일을 다운로드하지 않고, 점진적으로 데이터를 가져올지 여부를 결정한다.
이 모드로 설정하면 반복자(Iterator) 형식으로 데이터를 반환해 점진적으로 다운로드 할 수 있다.

10. num_proc: 로컬에서 데이터세트를 다운로드하고 생성할 대 사용할 프로세스 수를 설정한다.

Llama3 학습 파라미터 설정

model_name: "meta-llama/Meta-Llama-3.1-8B-Instruct"
dataset_path: "."
max_seq_length: 512
output_dir: "./llama-3.1-korean-8b-hf-20-epoch"

learning_rate: 0.00005
lr_scheduler_type: "linear"
num_train_epochs: 5

gradient_accumulation_steps: 4
optim: "adamw_torch_fused"

1. model_name
사용하고자하는 모델의 ID를 나타내며, 허깅페이스에 등록된 모델 이름을 적는다.

2. dataset_path
학습에 사용할 데이터셋이 위치한 경로를 지정한다.

3. max_seq_length
모델이 처리할 수 있는 최대 시퀀스 길이를 설정하는 파라미터이다. 이 파라미터 값을 모델이 한 번에 처리할 수 있는 입력 텍스트의 최대 길이를 결정하며, 더 긴 시퀀스는 자동으로 max_seq_length 길이로
잘리고, 더 짧은 시퀀스는 패딩된다. 이 설정은 모델의 성능과 메모리 사용량에 직접적인 영향을 미치므로 사용 가능한 하드웨어 자원과 작업의 특성을 고려해 적절히 조정해야 한다.

4. output_dir
모델 학습 과정에서 생성되는 학습된 모델, 학습 중간중간에 저장되는 체크 포인트 파일들을 저장할 디렉터리이다.

5. report_to
모델 학습 과정에서 발생하는 로그와 성능 지표를 어떤 플랫폼에 기록할지 지정하는 옵션이다. 위에서는 wandb를 사용한다.
이러한 도구들은 학습 손실, 학습률 변화 등의 중요한 지표들을 실시간 모니터링이 가능하게 도와준다.

6. learning_rate
신경망의 가중치를 얼마나 크게 조정할지 결정하는 중요한 하이퍼파아미터이다. 일반적으로 전체 파인 튜닝은 1e-5에서 5e-5 사이의 값이 자주 사용되며, 이번 학습에서는 0.00005를 사용할 예정이다.
학습률의 선택은 모델의 크기, 데이터셋의 특성, 학습 목표 그리고 학습 방법에 따라 달라진다. 높은 학습률은 최적점을 빠르게 찾을 수 있지만, 학습이 불안정해지거나 발산할 위험이 있다.
반면, 너무 낮은 학습률은 안정적이지만 학습이 매우 느리거나 지역 최적점에 갇힐 수 있다.

7. lr_scheduler_type
학습률 스케줄러이며, 학습 과정에서 학습률의 변화를 관리한다.
constant로 설정하면 학습 전체에 걸쳐 일정한 학습률을 유지한다. 다른 옵션으로는 linear, cosine, consine_with_restarts 등이 있으며, 각각 다른 방식으로 학습률을 조절한다.

8. num_train_epochs
에폭이며 전체 데이터셋을 한 번 학습하는 주기를 의미한다.
1로 설정하면 데이터셋을 한 번만 학습한다. 에폭 수를 늘리면 모델이 데이터를 더 많이 학습하지만, 과적합의 위험도 증가할 수 있다.

9. per_device_train_batch_size
각 GPU 또는 CPU에서 한 번에 처리할 수 학습 데이터의 개수이다.
일반적으로 컴퓨터의 메모리와 처리 능력이 감당할 수 있는 가장 큰 배치 크기를 사용하면 더 빠르게 학습 할 수 있다.

10. per_device_eval_batch_size
평가 시 각 디바이스에서 처리할 데이터의 개수이다.

11. gradient_accumulation_steps
그레이디언트 누적 기법을 사용해 가상의 더 큰 배치 크기 효과를 얻는 방법이다.
위에서는 4로 설정했고, 이는 모델이 4번의 순전파와 역전파를 수행한 후 한 번씩 가중치를 업데이트 한다. 즉 4개의 작은 배치에서 계산된 그레이디언트를 모아 한 번에 적용하는 것이다.
이 기법은 GPU 메모리 사용량을 줄이면서도 큰 배치 크기의 효과를 얻을 수 있어 메모리 효율성이 높다.
실제 배치 크기에 누적 스텝 수를 곱한 만큼의 유효 배치 크기 효과를 얻을 수 있어 더 안정적인 학습이 가능하다.
이는 GPU 메모리가 제한적이거나 매우 큰 모델을 학습할 때 특히 유용하지만, 학습 시간이 약간 늘어날 수 있다는 점을 고려해야한다.

12.optim
모델 학습에 사용할 최적화 알고리즘을 지정한다.
위에서는 adamw_torch_fused를 사용한다. 이는 파이토치에서 제공하는 최적화된 AdamW 구현한다. AdamW는 널리 사용되는 Adam 옵티마이저의 변형으로, 적응적 학습률 조정과 모멘텀을 결합한 방식이다.
여기에 가중치 감쇠(weight decay)를 효과적으로 적용해 과적합을 줄이는 특징이 있다. fused 버전은 이러한 AdamW의 장점을 유지하면서도 계산 효율을 높인 것이다.
이 옵티마이저는 여러 연산을 하나로 합쳐(fuse) GPU에서 더 빠르게 실행될 수 있도록 최적화됐다.

13.logging_steps
로그를 기록할 주기를 스텝 단위로 설정한다. 10으로 설정하면 10스텝마다 학습 진행 상황과 성능 지표가 기록된다.

14.save_strategy
모델 체크포인트를 저장하는 전략을 설정한다. epoch으로 설정하면 각 에폭이 끝날때 마다 모델을 저장한다.
다른 옵션으로는 steps(특정 스텝마다 저장) 또는 no(저장하지 않음) 등이 있다.

15.weight_decay
과적합을 방지하고 모델의 일반화 성능을 향상시키는 데 사용된다.

16.max_grad_norm
그레이디언트 클리핑의 임계값을 지정한다.
그레이디언트 클리핑은 학습 과정에서 그레이디언트의 크기가 지나치게 커지는 것을 방지하는 기법이다.
0.3이라는 값은 모든 파라미터의 그레이디언트를 합한 노름이 0.3을 초과할 경우, 그 비율에 맞춰 그레이디언트를 축소한다.
이는 급격한 가중치 변화를 억제해 학습의 안정성을 높이고 그레이디언트 폭발 문제를 완화한다.

17.warmup_ratio
학습률 웜업 기법을 제어한다. 웜업은 학습 초기에 학습률을 매우 작은 값에서 시작해 점진적으로 목표 학습률까지 증가시키는 방법이다.
0.03이라는 값은 전체 학습 스텝의 3% 동안 이 웜업 과정이 진행됨을 의미한다.
이 기법의 주요 목적은 학습 초기의 급격한 파라미터 변화를 방지하고, 모델이 안정적으로 학습을 시작할 수 있게 하는 것이다.
초기 학습 단계에서의 그레이디언트 불안정성을 줄이고, 모델이 더 나은 지역 최적점으로 수렴할 가능성을 높인다.
그러나 최적의 웜업 비율은 모델 크기, 데이터셋 특성, 전체 학습 에폭 수 등에 따라 달라질 수 있으며, 실험을 통해 조정이 필요할 수 있다.

18.bf16
bfloat16 정밀도를 사용하도록 활성화한다. bfloat16은 32비트 부동소수점(float32)의 지수 부분은 그대로 유지하면서 가수 부분을 줄인 16비트 형식이다.
이는 넓은 동적 범위를 유지하면서 메모리 사용량을 절반으로 줄이고 연산 속도를 높인다. 특히 딥러닝 모델 학습에서 float32와 유사한 수치 안정성을 제공하면서 더 효율적인 학습을 가능하게 한다.

19.tf32
NVDIA의 TensorFloat-32(TF32) 형식을 활성화한다. TF32는 NVDIA의 Ampere 아키텍처 이상의 GPU에서 제공되는 특별한 내부 표현 방식이다. 이는 19비트 부동소수점 형식으로,
float32의 정밀도와 float16의 속도 사이의 균형을 제공한다. 이를 통해 프로그램의 변경 없이도 행렬 곱셈과 같은 주요 연산의 속도를 크게 향상할 수 있다.

20.gradient_checkpointing
그레이디언트 체크포인팅 기법을 활성화한다. 이는 대규모 신경망 모델 학습 시 메모리 사용을 최적화하는 고급 기술이다.
일반적으로 역전파 과정에서 모든 중간 활성화 값을 메모리에 저장하는 대신, 그레이디언트 체크포인팅은 전략적으로 선택된 일부 활성화만 저장한다.
나머지 활성화는 역전파 중에 필요할 때 재계산된다. 이 방식으로 메모리 사용량을 크게 줄이는 대신 약간의 계산 오버헤드를 발생시킨다.
결과적으로 더 큰 배치 크기를 사용하거나, 더 깊은 모델을 학습하거나 또는 더 긴 시퀀스를 처리할 수 있게 된다.
한편, gradient_checkpointing을 사용하다 보면 use_cache와 함께 쓸 수 없다는 경고문이 뜬다.
use_cache는 모델의 키와 값을 캐시해 추론 시 계산 효율을 높이는 기능이다.
이는 주로 텍스트 생성 작업에서 사용되며, 이전에 계산된 키와 값을 재사용해 연속적인 토큰 생성 속도를 높인다.
그러나 gradient_checkpointing과 use_cache는 동시에 사용할 수 없다. 그 이유는 gradient_checkpointing이 메모리 절약을 위해 중간 활성화를 재계산하는 반면, use_cache는 이러한 중간 결과를 저장하고
재사용하기 때문이다. 두 기술의 작동 방식이 서로 상충되어 함께 사용할 경우 오류가 발생하거나 의도한 효과를 얻을 수 없다.

21. fsdp
full_shard auto_wrap는 Fully Sharded Data Pararallel(FSDP)을 활성화해 모델을 여러 GPU에 분산시켜 메모리 효율을 높이는 기술을 적용한다.
full_shard는 모든 파라미터를 분산시키고, auto_wrap은 자동으로 모델을 FSDP로 래핑한다.
FSDP란 각 GPU가 전체 모델의 일부만을 저장하고 필요할 때 다른 GPU와 통신해 데이터를 주고받는 방식으로 작동한다.

22. fsdp_config
fsdp_config의 하위 속성 설정들은 이 분산 처리의 효율성을 더욱 높여준다.

23. backward_prefetch
backward_pre 설정은 FSDP에서 역전파 과정을 최적화하는 중요한 기능이다.
이 설정은 역전파 계산이 시작되기 직전에 필요한 모델 파라미터를 미리 가져오는 전략을 활성화한다.
FSDP에서 모델 파라미터가 여러 GPU에 분산돼 있기 때문에 이 설정은 각 역전파 단계 시작 전에 필요한 파라미터를 효율적으로 준비한다.
이를 통해 대기 시간이 감소하고, 파라미터를 가져오는 통신 작업과 실제 계산 작업을 부분적으로 병렬화할 수 있으며, 메모리 사용을 최적화한다.

24. forward_prefetch
forward_prefetch를 false로 설정하면 순전파 과정 중 추가적인 데이터 프리페치(미리 가져오기)를 비활성화한다.
일반적으로 프리페치는 성능을 향상시키기 위해 사용되지만, FSDP 컨텍스트에서는 오히려 불필요할 수 있다.
순전파 중 프리페치를 비활성화함으로써 모델은 현재 필요한 데이터만을 처리하게 된다.
이는 GPU 간 불필요한 데이터 전송을 줄이고, 메모리 사용을 최적화하며, 전체적인 계산 효율을 높일 수 있다.

25. use_orig_params
use_orig_params를 false로 설정하면 원본 파라미터 대신 분할된 파라미터를 직접 사용해 메모리 효율성을 극대화한다.
