RAG는 크게 **인덱싱**과 **검색+생성** 두 단계로 나뉩니다. 각 단계를 깊이 파헤쳐 보겠습니다.인덱싱이 완료되면 실제 질문이 들어왔을 때 **검색+생성** 단계가 실시간으로 실행됩니다.마지막으로, 일반 LLM과 RAG의 근본적인 차이를 비교해보겠습니다.

### 전체 흐름 요약

RAG의 핵심 메커니즘은 두 단계로 나뉩니다.

**인덱싱 단계 (사전 처리)** — 외부 문서를 불러와 500토큰 내외로 청킹하고, 임베딩 모델로 벡터로 변환해 Vector DB에 저장합니다. 이 과정은 서비스 시작 전 한 번만 수행하거나, 문서 업데이트 시 증분 갱신합니다.

**검색+생성 단계 (실시간)** — 사용자 질문이 들어오면 동일한 임베딩 모델로 질문을 벡터화하고, Vector DB에서 코사인 유사도 기준 상위 K개 청크를 검색합니다. 검색된 청크를 프롬프트에 `[Context]`로 주입한 뒤 LLM에 전달하면, LLM은 자신의 학습 데이터가 아닌 주입된 문맥을 근거로 답변을 생성합니다.

이 구조 덕분에 LLM을 재학습하지 않고도 실시간 정보, 사내 기밀 문서, 도메인 특화 지식을 LLM에 즉시 활용할 수 있습니다. 각 카드를 클릭하면 해당 주제에 대해 더 자세한 설명을 받을 수 있습니다.

### RAG의 한계점과 개선 방법

**청킹 품질 문제** — 문맥이 잘리면 검색 정확도 저하 → 재귀적 청킹 + 오버랩(overlap) 적용

**임베딩 모델 한계** — 모델이 도메인에 맞지 않으면 유사도 계산 부정확 → 도메인 특화 임베딩 파인튜닝

**Top-K 선택 오류** — 관련 없는 문서가 상위에 오는 경우 → Re-ranking 모델(Cross-encoder) 추가

**컨텍스트 윈도우 초과** — 너무 많은 청크 주입 시 LLM 한계 도달 → 청크 수(K) 조절, 압축 기법 적용

**HyDE 기법** — 질문 대신 가상 답변을 임베딩해 검색 정확도를 높이는 고급 기법
