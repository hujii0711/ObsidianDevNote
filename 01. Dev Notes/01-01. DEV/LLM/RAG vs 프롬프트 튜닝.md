## RAG vs 프롬프트 튜닝 — 개념적 위치

AI 시스템을 개선하는 방법은 크게 세 층위로 나뉩니다:

```
┌─────────────────────────────────────┐
│         Fine-tuning (파인튜닝)        │  ← 모델 가중치 자체를 변경
├─────────────────────────────────────┤
│         Prompt Engineering           │  ← 입력 텍스트를 설계
│    (프롬프트 엔지니어링 / 튜닝)         │
├─────────────────────────────────────┤
│              RAG                    │  ← 외부 데이터를 검색 후 주입
└─────────────────────────────────────┘
```

RAG는 프롬프트에 문맥을 **동적으로 삽입**하므로 프롬프트 엔지니어링과 겹치는 면이 있습니다. 하지만 RAG를 단순히 프롬프트 튜닝의 한 기법이라고 보기 어려운 이유가 있습니다.

---

## RAG가 프롬프트 튜닝과 다른 점

**프롬프트 튜닝/엔지니어링**은

- 프롬프트 문장 자체를 잘 설계하는 행위
- 입력은 **정적(static)**
- "역할을 부여하거나", "예시를 넣거나", "출력 형식을 지정하는" 것

**RAG**는

- 질문마다 외부 DB/문서를 **실시간 검색**해서 문맥을 가져옴
- 입력이 **동적(dynamic)** — 질문이 바뀌면 검색 결과도 바뀜
- 검색 시스템(벡터 DB, 임베딩 모델)이라는 **별도 인프라**가 필요

즉, RAG는 **"검색 시스템 + 프롬프트 조립 + LLM 추론"** 이 결합된 **아키텍처**입니다.

---

## 정확한 분류

|관점|분류|
|---|---|
|모델을 건드리는가?|❌ 아니요 → Fine-tuning이 아님|
|프롬프트를 조작하는가?|✅ 예 → 프롬프트 엔지니어링 요소 포함|
|외부 시스템이 필요한가?|✅ 예 → 단순 프롬프트 기법을 초과|
|가장 정확한 분류|**"검색 증강 추론 아키텍처"**|

---

## 한 줄 요약

> RAG는 프롬프트에 동적 문맥을 주입한다는 점에서 프롬프트 엔지니어링과 **교집합**이 있지만, 검색 인프라와 결합된 **독립적인 시스템 아키텍처**로 보는 것이 더 정확합니다.
