겸사겸사 제가 만든 스킬 샘플 공유 드려요

너는 senior performance engineer 겸 backend engineer다.

목표:  
PostgreSQL 기반 멀티테넌트 서비스의 오픈 전 성능 테스트 체계를 repo 안에 구현한다.  
예상 누적 데이터는 10억 건이다.  
테스트는 로컬 환경에서 실행한다.  
운영 DB가 아니라 k6 테스트 전용 복제 PostgreSQL DB를 사용한다.  
k6 기반으로 smoke, average-load/load, stress, spike, soak, breakpoint 테스트를 모두 구성한다.  
API 경유 테스트를 주 테스트로 만들고, PostgreSQL 직접 테스트는 쿼리 진단용 보조 테스트로만 만든다.

전체 테스트 구조:  
- k6 → local API server → cloned PostgreSQL DB  
- optional: k6 → cloned PostgreSQL DB direct, only for DB smoke/baseline diagnostics

중요한 안전 규칙:  
1. 운영 DB, production BASE_URL, production tenant를 절대 기본값으로 사용하지 마라.  
2. 모든 테스트는 로컬 API 서버와 k6 테스트 전용 복제 DB를 대상으로 한다.  
3. production으로 보이는 URL, DB host, DB name, tenant id가 감지되면 즉시 실패하도록 guard를 넣어라.  
4. DB credential, token, API key, tenant id를 코드에 하드코딩하지 마라.  
5. 모든 민감값은 ENV 또는 .env.example로만 표현하라.  
6. write 테스트는 ALLOW_WRITE_TESTS=true 없이는 실행되지 않게 하라.  
7. DB 직접 테스트는 ALLOW_DB_DIRECT_TESTS=true 없이는 실행되지 않게 하라.  
8. 복제 DB라도 원본 운영 데이터가 포함될 수 있으므로 PII/민감정보를 로그나 리포트에 출력하지 마라.  
9. tenant 실제 이름, 고객명, 이메일, 전화번호, 식별 가능한 값은 리포트에 남기지 마라.  
10. 대규모 DELETE, TRUNCATE, VACUUM FULL, REINDEX, schema migration은 자동 실행하지 마라.  
11. EXPLAIN ANALYZE로 INSERT/UPDATE/DELETE/MERGE를 분석할 경우 반드시 BEGIN; ... ROLLBACK; 안에서만 실행되도록 하라.  
12. 불확실한 endpoint, query, tenant 구조, 인증 방식은 추측하지 말고 TODO로 남겨라.

테스트 대상:  
- 서비스 유형: 멀티테넌트  
- DB: PostgreSQL  
- 실행 위치: local  
- DB 대상: k6 테스트용 복제 PostgreSQL DB

- 테스트 도구: k6  
- DB 직접 테스트 도구: xk6-sql + PostgreSQL driver 사용 가능하면 구성, 불가능하면 문서화만 수행

scenario별 요구사항:

1. smoke.js  
목적:  
- 로컬 API 서버, 인증, DB 연결, 핵심 read/write path가 최소 부하에서 정상인지 확인

구성:  
- 낮은 VU 또는 낮은 iteration  
- health check  
- auth check  
- 핵심 목록 조회  
- 핵심 상세 조회  
- large/medium/small tenant 각각 최소 1회 요청  
- write 허용 시 작은 create/update/readback  
- write 비허용 시 write skip 로그 표시

성공 조건:  
- 5xx = 0  
- check success = 100%  
- tenant context 정상  
- p95/p99 threshold 통과

2. tenant-isolation.smoke.js  
목적:  
- 멀티테넌트 데이터 격리 검증

구성:  
- tenant A token/context로 tenant B 데이터 접근 시도  
- tenant_id 누락 요청  
- 잘못된 tenant_id 요청  
- 다른 tenant object_id 접근  
- list response에 다른 tenant 데이터가 섞이는지 확인  
- cache key isolation 의심 endpoint 확인

성공 조건:  
- cross-tenant access는 403/404 또는 서비스 정책상 올바른 거절 응답  
- 다른 tenant 데이터 노출 0  
- tenant isolation failure rate = 0

3. load.average.js  
목적:  
- 예상 평균 RPS에서 SLA 만족 여부 확인

구성:  
- constant-arrival-rate 또는 ramping-arrival-rate 사용  
- AVG_RPS 기준  
- 30~60분 실행 가능하도록 구성  
- 실제 사용자 journey 비율 적용  
- read/write 비율 적용  
- tenant traffic weight 적용

tenant traffic model 예시:  
- large tenant: 전체 요청의 50~70%  
- medium tenant: 전체 요청의 20~40%  
- small tenant: 전체 요청의 5~20%

성공 조건:  
- error rate threshold 통과  
- p95/p99 threshold 통과  
- tenant_class별 latency 확인 가능  
- endpoint별 latency 확인 가능  
- 리포트 저장

4. stress.js  
목적:  
- peak 이상에서 병목과 graceful degradation 확인

구성:  
- ramping-arrival-rate 사용  
- stages:  
  - 0.5x peak  
  - 1.0x peak  
  - 1.2x peak  
  - 1.5x peak  
  - 2.0x peak  
  - ramp down  
- 대형 tenant 중심 부하 포함  
- 중/소형 tenant 요청도 일부 유지

성공 조건:  
- 1.0x peak에서 SLA 통과  
- 1.5x peak에서 error rate 허용 범위 확인  
- 2.0x peak에서 장애 전파, retry storm, DB connection exhaustion 여부 관측  
- 단계별 결과를 summary에 기록

5. spike.js  
목적:  
- 오픈 직후, 마케팅, 알림 발송 같은 순간 폭증 대응 확인

구성:  
- 0 또는 low traffic에서 3x peak까지 급상승  
- 짧은 유지  
- 급하강  
- 특정 large tenant에 spike 집중  
- small tenant observer 요청 유지

확인:  
- autoscaling 전 구간 실패율  
- cache stampede 의심  
- DB connection pool 고갈  
- timeout 급증  
- spike 이후 회복 여부

성공 조건:  
- 5xx 폭증 없음  
- recovery 이후 latency 정상화  
- observer tenant가 과도하게 영향받지 않음

6. noisy-neighbor.js  
목적:  
- 대형 tenant 부하가 다른 tenant에 영향을 주는지 확인

구성:  
- noisy tenant: large tenant 1개에 고부하  
- observer tenant: small tenant 1~3개에 낮은 지속 요청  
- noisy tenant와 observer tenant metric을 분리  
- tenant_class, role=noisy|observer tag 사용

성공 조건:  
- observer tenant p95/p99가 기준 이상으로 악화되지 않음  
- observer tenant error rate 증가 없음  
- tenant isolation 유지  
- noisy tenant 장애가 전체 장애로 번지지 않음

7. soak.js  
목적:  
- 장시간 운영 중 latency 우상향, memory leak, connection leak, slow query 누적, lock 누적 확인

구성:  
- SOAK_DURATION ENV 기반  
- 기본 문서 권장: 6h, 12h, 24h  
- 평균~피크 60~100% 사이 RPS를 ENV로 받음  
- mixed tenant traffic  
- read/write 비율 반영  
- 테스트 전/중/후 PostgreSQL snapshot 수집 스크립트와 연계

성공 조건:  
- latency 우상향 없음  
- error rate threshold 통과  
- DB connection 누수 없음  
- slow query 누적 없음  
- lock wait 누적 없음  
- replication lag 누적 없음  
- tenant별 성능 편차 기록

8. breakpoint.js  
목적:  
- 시스템 한계점 탐색

구성:  
- BREAKPOINT_START_RPS에서 시작  
- BREAKPOINT_STEP_RPS만큼 단계 증가  
- 각 단계는 BREAKPOINT_STEP_DURATION 유지  
- threshold 실패 지점과 saturation 지점 기록

성공 조건:  
- pass/fail보다 breaking point를 정확히 리포트  
- 최초 p95/p99 급증 지점 기록  
- 최초 error rate 급증 지점 기록  
- DB CPU/connection/lock/IO saturation 의심 지점 기록

9. db.smoke.js  
목적:  
- k6 테스트용 복제 DB에 직접 연결해 DB 기본 상태 확인

조건:  
- ALLOW_DB_DIRECT_TESTS=true일 때만 실행

구성:  
- SELECT 1  
- current_database()  
- PostgreSQL version  
- 핵심 테이블 existence check  
- large/medium/small tenant별 최소 데이터 존재 확인  
- 대표 read query 1~3개  
- write 허용 시 BEGIN; write; readback; ROLLBACK;

성공 조건:  
- DB 연결 성공  
- 핵심 테이블 접근 가능  
- tenant fixture에 해당하는 데이터 존재  
- 대표 read query 성공  
- DB error rate = 0

10. db.baseline.js  
목적:  
- 대표 PostgreSQL query latency baseline 측정

조건:  
- ALLOW_DB_DIRECT_TESTS=true일 때만 실행  
- API 성능 판단용이 아니라 DB query 진단용임을 README에 명시

구성:  
- 대표 read query latency 측정  
- 대표 write query는 rollback 또는 전용 test table만 사용  
- large/medium/small tenant별 query latency 분리  
- custom Trend metric 기록  
- full table scan 유발 가능성이 있는 쿼리는 기본 비활성화

10억 건 복제 DB 준비 문서 요구사항:

perf/postgres/[README.md](README.md)에 다음 내용을 작성하라.

- 테스트 DB는 운영 DB가 아니라 k6 전용 복제 DB여야 한다.  
- 운영 데이터 dump를 사용할 경우 반드시 비식별화/sanitization이 필요하다.  
- 10억 건 숫자만 맞춘 더미 데이터는 부족하다.  
- 실제와 유사한 분포가 중요하다.

반드시 반영할 분포:  
- tenant별 데이터 skew  
- tenant별 traffic skew  
- large / medium / small tenant  
- tenant 내부 hot/warm/cold data  
- created_at / updated_at 시간 분포  
- status/category 선택도  
- user/object hotspot  
- pagination 깊이  
- 검색 대상 컬럼 길이/중복도  
- read/write 비율  
- write hotspot

위험 패턴 체크리스트:  
- tenant_id 없는 query  
- tenant_id 없는 cache key  
- offset pagination  
- count(*) 남발  
- 낮은 선택도 index  
- LIKE '%keyword%'  
- N+1 query  
- 대량 join  
- 대량 update/delete  
- hot row update  
- long transaction  
- seq scan  
- sort spill  
- autovacuum 지연  
- index bloat

Definition of Done:

공통 골인:  
- production guard 존재  
- secret 하드코딩 없음  
- ENV 누락 시 명확히 실패  
- threshold 존재  
- JSON/summary report 저장  
- README 실행법 존재  
- tenant_class별 metric 존재  
- endpoint/journey별 metric 존재

Smoke 골인:  
- local API 정상  
- cloned DB 연결 정상  
- auth 정상  
- large/medium/small tenant read path 정상  
- write 허용 시 최소 write-readback 정상  
- 5xx = 0  
- checks = 100%

Tenant isolation 골인:  
- cross-tenant access 차단  
- tenant_id 누락/변조 요청 차단  
- 다른 tenant 데이터 노출 0  
- cache isolation 의심 없음

Average-load 골인:  
- AVG_RPS 기준 실행  
- 사용자 journey 비율 반영  
- tenant traffic skew 반영  
- p95/p99/error threshold 통과  
- tenant_class별 p95/p99 확인 가능

Stress 골인:  
- peak 1.2x/1.5x/2.0x 단계별 결과 기록  
- saturation 지점 기록  
- graceful degradation 여부 기록  
- 대형 tenant 부하가 전체 장애로 번지는지 확인

Spike 골인:  
- 0 또는 low traffic에서 3x peak 급상승  
- recovery 구간 검증  
- spike 이후 latency/error 정상화 여부 기록  
- observer tenant 영향 기록

Noisy-neighbor 골인:  
- noisy tenant와 observer tenant metric 분리  
- 대형 tenant 부하가 소형 tenant에 미치는 영향 확인  
- observer tenant latency/error 기준 충족

Soak 골인:  
- 6h/12h/24h 실행 가능  
- latency 우상향 여부 확인 가능  
- connection leak 확인 가능  
- slow query/lock/replication lag 관측 절차 포함  
- PostgreSQL snapshot 전/중/후 비교 가능

Breakpoint 골인:  
- RPS 증가에 따른 최초 실패 지점 기록  
- p95/p99 급증 지점 기록  
- error rate 급증 지점 기록  
- DB 병목 의심 지표 기록

DB smoke/baseline 골인:  
- ALLOW_DB_DIRECT_TESTS=true 없이는 실행 불가  
- SELECT 1과 대표 read query 검증  
- write는 rollback 또는 test table만 사용  
- query별 latency metric 기록  
- tenant별 query latency 분리
