## 클로드 코드 + 하네스 엔지니어링 핵심 팁

### 🧠 하네스 엔지니어링이란?

AI 모델 자체를 개선하는 게 아니라, <font color="#ff0000">모델이 일할 환경(도구, 루프, 상태 관리)을 잘 설계</font>하는 것입니다. "하네스는 Claude를 똑똑하게 만드는 게 아니라, Claude에게 손과 눈과 작업 공간을 준다"는 개념입니다.

---

### 1. Initializer Agent + Coding Agent 분리

첫 번째 세션에는 **초기화 에이전트(Initializer Agent)**를 사용해 환경을 세팅하고, 이후 모든 세션은 **코딩 에이전트(Coding Agent)**가 점진적으로 작업을 이어나가도록 구조를 나눕니다.

- **Initializer**: `init.sh`, `claude-progress.txt`, feature list JSON, 초기 git commit 생성
- **Coding Agent**: 매 세션마다 progress 파일 읽고 → 다음 기능 선택 → 구현 → git commit

---

### 2. Feature List를 JSON으로 관리

기능 요구사항을 JSON 파일로 작성하고, 모든 기능을 초기에 `"passes": false`로 표시합니다. 코딩 에이전트는 이 파일에서 상태값만 변경하도록 강하게 지시합니다. Markdown보다 JSON이 에이전트가 임의로 수정/삭제할 가능성이 낮습니다.

```json
{
  "description": "New chat button creates a fresh conversation",
  "passes": false
}
```

---

### 3. 한 번에 하나의 기능만 구현 (Incremental Progress)

<font color="#ff0000">에이전트는 한 번에 하나의 기능만 작업하도록 프롬프트해야</font> 합니다. 
이 점진적 접근법은 에이전트가 한꺼번에 너무 많이 하려는 경향을 방지하는 데 결정적이었습니다.

---

### 4. Git을 안전망으로 활용

매 성공한 작업 후 git commit을 하세요. 복구는 언제든 `git reset` 하나로 가능합니다. 에이전트가 잘못된 코드 변경 시 즉시 되돌릴 수 있습니다.

---

### 5. 별도 Evaluator 세션으로 품질 검증

에이전트는 자신의 작업을 너무 후하게 평가합니다. 새로운 컨텍스트에서 회의적인 평가자(Evaluator)를 별도로 실행하면 정직한 피드백을 받을 수 있습니다. 실패 시 최대 2번 재시도 후 종료하여 무한 루프를 방지하세요.

---

### 6. 브라우저 자동화로 E2E 테스트

Claude가 기능을 완료로 표시하기 전에 Puppeteer MCP 같은 브라우저 자동화 도구로 실제 사용자처럼 테스트하도록 명시적으로 지시해야 합니다. 이 방식이 코드만 보는 것으로는 발견하기 어려운 버그를 찾아내는 데 크게 도움이 됐습니다.

---

### 7. 세션 시작 루틴 고정

매 코딩 세션은 아래 순서로 시작하도록 프롬프트합니다: `pwd` 확인 → git 로그와 progress 파일 읽기 → feature list에서 다음 미완성 기능 선택. 이렇게 하면 에이전트가 상황 파악에 토큰을 낭비하지 않습니다.

---

### 8. CLAUDE.md는 간결하게

<font color="#ff0000">CLAUDE.md는 60줄 이하로 유지</font>하세요. MCP 서버는 파일 I/O와 bash 명령 외의 기능을 에이전트에게 확장해주는 용도로 주로 활용합니다.
