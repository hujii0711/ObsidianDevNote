## 1. 전체 파이프라인 개요

```
개발자 로컬                    GitHub                          배포 서버 (M4 Mac)
─────────────                ────────                        ──────────────────
코드 작성                     Push / PR 생성
    │                            │
린트 + 포맷 확인                 ▼
    │                   ┌─────────────────┐
로컬 테스트 실행         │  GitHub Actions  │
    │                   │  CI Pipeline     │
    ▼                   │                  │
git push               │  1. Lint (ruff)  │
──────────────────→     │  2. Type Check   │
                        │  3. Unit Tests   │
                        │  4. Build Check  │
                        └────────┬────────┘
                                 │
                          Pass ──┤── Fail → PR 블로킹
                                 │
                                 ▼
                        ┌─────────────────┐
                        │  PR Merge       │
                        │  → main 브랜치   │
                        └────────┬────────┘
                                 │
                                 ▼
                        ┌─────────────────┐
                        │  CD Pipeline     │
                        │  (self-hosted)   │
                        │                  │
                        │  1. Pull latest  │
                        │  2. Install deps │
                        │  3. Run tests    │
                        │  4. Restart svc  │
                        └─────────────────┘
```

## 2. Git 브랜치 전략

### 2.1 브랜치 구조 (GitHub Flow 기반)

```
main (보호됨)
  │   ← 항상 배포 가능한 상태 유지
  │   ← PR 머지만 허용, 직접 push 금지
  │
  ├── feature/rag-reranker        ← 기능 개발
  ├── feature/streaming-sse       ← 기능 개발
  ├── fix/tokenizer-encoding      ← 버그 수정
  ├── data/add-rental-cases       ← 데이터 추가
  └── experiment/solar-model      ← 실험 (머지 안 할 수도 있음)
```

### 2.2 브랜치 네이밍 규칙

| 접두사 | 용도 | 예시 |
|--------|------|------|
| `feature/` | 새 기능 개발 | `feature/chat-history-api` |
| `fix/` | 버그 수정 | `fix/rag-empty-result-handling` |
| `refactor/` | 리팩터링 | `refactor/engine-singleton` |
| `data/` | 데이터 추가/수정 | `data/add-civil-law-statutes` |
| `test/` | 테스트 추가 | `test/api-edge-cases` |
| `experiment/` | 모델/파라미터 실험 | `experiment/llama3-comparison` |
| `docs/` | 문서 수정 | `docs/api-usage-guide` |

### 2.3 커밋 메시지 규칙

```
<type>(<scope>): <subject>

# type 종류
feat:     새 기능
fix:      버그 수정
refactor: 리팩터링 (기능 변화 없음)
test:     테스트 추가/수정
docs:     문서 변경
chore:    빌드, 설정 변경
data:     데이터 추가/전처리
perf:     성능 개선

# scope 종류
api, rag, llm, training, ui, data, config, ci

# 예시
feat(rag): 판례 문서 조항 단위 청킹 파이프라인 추가
fix(api): 빈 RAG 결과 시 500 에러 수정
test(rag): 리랭킹 정확도 테스트 추가
data(training): 임대차 상담 Q&A 2,000건 추가
perf(llm): 스트리밍 첫 토큰 지연 시간 개선
```

### 2.4 PR 규칙

```
PR 제목: 커밋 메시지와 동일한 형식
PR 본문:
  ## 변경 사항
  - 변경 내용 요약
  
  ## 테스트
  - [ ] 로컬 테스트 통과
  - [ ] 관련 테스트 추가/수정
  
  ## 리뷰 포인트
  - 리뷰어가 집중적으로 봐야 할 부분

머지 조건:
  ✅ CI 파이프라인 전체 통과
  ✅ 1명 이상 리뷰 승인 (팀인 경우)
  ✅ 충돌 없음
```

## 3. 로컬 개발 환경

### 3.1 개발 워크플로

```bash
# 1. 최신 main 동기화
git checkout main
git pull origin main

# 2. 기능 브랜치 생성
git checkout -b feature/my-feature

# 3. 코드 작성 ...

# 4. 린트 + 포맷 확인
ruff check src/ tests/
ruff format --check src/ tests/

# 5. 자동 포맷 적용
ruff format src/ tests/

# 6. 로컬 테스트
pytest tests/ -v

# 7. 커밋
git add src/rag/retriever.py tests/test_rag.py
git commit -m "feat(rag): 리랭킹 스코어 임계값 필터 추가"

# 8. 푸시 + PR 생성
git push -u origin feature/my-feature
gh pr create --title "feat(rag): 리랭킹 스코어 임계값 필터 추가"
```

### 3.2 pre-commit 훅 설정

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.6.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
```

```bash
# 설치
pip install pre-commit
pre-commit install

# 이후 git commit 시 자동으로 린트/포맷 실행
# 위반 시 커밋 차단
```

### 3.3 로컬 테스트 실행

```bash
# 전체 테스트
pytest tests/ -v

# 특정 모듈
pytest tests/test_api.py -v
pytest tests/test_rag.py -v
pytest tests/test_prompt.py -v

# 커버리지 측정
pytest tests/ --cov=src --cov-report=term-missing

# 빠른 실행 (LLM 모킹 테스트만)
pytest tests/ -v -k "not integration"
```

## 4. GitHub Actions CI 파이프라인

### 4.1 CI 워크플로 파일

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  PYTHON_VERSION: "3.11"

jobs:
  lint:
    name: Lint & Format
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: Install ruff
        run: pip install ruff

      - name: Run ruff check
        run: ruff check src/ tests/

      - name: Run ruff format check
        run: ruff format --check src/ tests/

  type-check:
    name: Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: Install dependencies
        run: |
          pip install --upgrade pip
          pip install pyright
          pip install -r requirements.txt

      - name: Run pyright
        run: pyright src/

  test:
    name: Tests
    runs-on: ubuntu-latest
    needs: [lint]
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: Cache pip packages
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}
          restore-keys: |
            ${{ runner.os }}-pip-

      - name: Install dependencies
        run: |
          pip install --upgrade pip
          pip install -r requirements.txt

      - name: Run tests
        run: pytest tests/ -v --tb=short

      - name: Run tests with coverage
        run: pytest tests/ --cov=src --cov-report=xml

      - name: Upload coverage
        uses: codecov/codecov-action@v4
        with:
          file: coverage.xml
        continue-on-error: true

  build-check:
    name: Build Check
    runs-on: ubuntu-latest
    needs: [test]
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t legal-llm:test .

      - name: Verify container starts
        run: |
          docker run -d --name test-container -p 8000:8000 legal-llm:test
          sleep 5
          docker logs test-container
          docker stop test-container
```

### 4.2 파이프라인 단계 설명

```
PR 생성/업데이트 시:

  ┌─────────┐     ┌────────────┐     ┌──────────┐     ┌─────────────┐
  │  Lint   │────→│ Type Check │────→│  Tests   │────→│ Build Check │
  │  (30초) │     │  (1분)     │     │  (2분)   │     │  (3분)      │
  └─────────┘     └────────────┘     └──────────┘     └─────────────┘
  
  ruff check      pyright           pytest           docker build
  ruff format     타입 오류 검출     단위 테스트       이미지 빌드
                                    커버리지 측정      컨테이너 기동 확인
```

### 4.3 CI에서 MLX 테스트 제외

```
MLX는 Apple Silicon 전용이므로 GitHub Actions (Linux)에서 실행 불가.
→ MLX 의존 테스트는 마커로 분리

# conftest.py
import pytest
import platform

mlx_available = platform.system() == "Darwin" and platform.machine() == "arm64"
requires_mlx = pytest.mark.skipif(
    not mlx_available,
    reason="MLX requires Apple Silicon"
)

# 테스트에서 사용
@requires_mlx
def test_mlx_model_loading():
    from mlx_lm import load
    model, tokenizer = load("mlx-community/EEVE-Korean-10.8B-4bit")
    assert model is not None
```

## 5. 테스트 전략

### 5.1 테스트 피라미드

```
              ┌──────────┐
              │ E2E 테스트 │  ← 수동 (Streamlit UI에서 실제 질의)
              │  (소수)    │
            ┌─┴──────────┴─┐
            │  통합 테스트    │  ← 로컬 전용 (실제 모델 + ChromaDB)
            │  (M4에서만)    │
          ┌─┴──────────────┴─┐
          │    단위 테스트      │  ← CI에서 실행 (모킹)
          │  (가장 많음)       │
          └──────────────────┘
```

### 5.2 테스트 카테고리

| 카테고리 | 실행 환경 | 모킹 여부 | 파일 패턴 |
|---------|----------|----------|----------|
| **단위 테스트** | CI + 로컬 | LLM 모킹, DB 모킹 | `test_*.py` |
| **통합 테스트** | 로컬(M4)만 | 실제 ChromaDB, 모킹 LLM | `test_*_integration.py` |
| **E2E 테스트** | 로컬(M4)만 | 전부 실제 | 수동 실행 |

### 5.3 단위 테스트 범위

```
src/api/
├── routes.py     → test_api.py
│   ├── POST /api/chat          정상 응답, 빈 메시지 거부, 모델 미로딩 503
│   ├── POST /api/chat/stream   SSE 이벤트 형식 검증
│   ├── GET  /api/history/{id}  정상 조회, 빈 세션
│   ├── POST /api/feedback      정상 저장, 잘못된 rating 거부
│   └── GET  /api/health        상태 응답 형식
├── schemas.py    → test_schemas.py (Pydantic 검증)
└── exceptions.py → test_api.py에 포함

src/rag/
├── indexer.py    → test_rag.py
│   ├── 문서 분류 (법조문/판례/용어/일반)
│   └── 법조문 조항 단위 분리
├── retriever.py  → test_rag.py (모킹)

src/llm/
├── prompt.py     → test_prompt.py
│   ├── 컨텍스트 있을 때 프롬프트 형식
│   ├── 컨텍스트 없을 때 "관련 자료를 찾지 못했습니다"
│   └── 시스템 프롬프트 포함 확인

src/training/
└── prepare_data.py → test_prepare_data.py
    ├── 유효/무효 데이터 검증
```

### 5.5 E2E 테스트 체크리스트 (수동)

```
서버 기동 후 Streamlit UI에서:

기본 기능:
  □ 질문 입력 → 응답 생성 확인
  □ 면책 조항이 모든 응답에 포함
  □ 참고 자료 표시 (RAG 사용 시)
  □ RAG 토글 OFF → 참고 자료 없이 응답
  □ 새 대화 시작 → 이력 초기화

엣지 케이스:
  □ 매우 긴 질문 (1000자 이상)
  □ 법률과 무관한 질문 → 적절히 거부/안내
  □ 빈 질문 → 에러 처리
  □ 연속 빠른 질문 → 큐잉 정상 동작

성능:
  □ 첫 토큰 출력까지 < 3초 (스트리밍)
  □ 전체 응답 < 30초
  □ RAG 검색 < 1초
```

## 6. CD (Continuous Deployment) 전략

### 6.1 배포 아키텍처

```
이 프로젝트는 M4 Mac 로컬 서버에 배포합니다.
클라우드가 아닌 self-hosted runner를 사용합니다.

GitHub Repository
    │
    │  main 머지 시 CD 트리거
    ▼
┌──────────────────────────┐
│  M4 Mac (self-hosted     │
│         GitHub Runner)   │
│                          │
│  1. git pull             │
│  2. pip install          │
│  3. pytest (최종 확인)    │
│  4. 서비스 재시작         │
│     - API 서버           │
│     - Streamlit UI       │
└──────────────────────────┘
```

### 6.2 Self-hosted Runner 설정

```bash
# M4 Mac에서 GitHub Actions Runner 설치

# 1. GitHub 리포지토리 > Settings > Actions > Runners > New self-hosted runner

# 2. 다운로드 및 설정
mkdir actions-runner && cd actions-runner
curl -o actions-runner-osx-arm64.tar.gz -L https://github.com/actions/runner/releases/download/v2.XXX/actions-runner-osx-arm64-2.XXX.tar.gz
tar xzf actions-runner-osx-arm64.tar.gz
./config.sh --url https://github.com/YOUR_ORG/LawLLM --token YOUR_TOKEN

# 3. 서비스로 등록 (백그라운드 실행)
./svc.sh install
./svc.sh start
```

### 6.3 CD 워크플로 파일

```yaml
# .github/workflows/cd.yml
name: CD - Deploy to M4 Mac

on:
  push:
    branches: [main]

jobs:
  deploy:
    name: Deploy
    runs-on: self-hosted  # M4 Mac Runner

    steps:
      - name: Pull latest code
        run: |
          cd /path/to/LawLLM
          git fetch origin main
          git reset --hard origin/main

      - name: Install dependencies
        run: |
          cd /path/to/LawLLM
          source .venv/bin/activate
          pip install -r requirements.txt

      - name: Run tests (final verification)
        run: |
          cd /path/to/LawLLM
          source .venv/bin/activate
          pytest tests/ -v --tb=short

      - name: Restart API server
        run: |
          # 기존 프로세스 종료
          pkill -f "uvicorn src.api.main:app" || true
          sleep 2

          # 새로 시작 (백그라운드)
          cd /path/to/LawLLM
          source .venv/bin/activate
          nohup uvicorn src.api.main:app \
            --host 0.0.0.0 --port 8000 \
            > logs/api.log 2>&1 &
          
          # 기동 확인 (헬스체크)
          sleep 30  # 모델 로딩 대기
          curl -sf http://localhost:8000/api/health || exit 1

      - name: Restart Streamlit UI
        run: |
          pkill -f "streamlit run" || true
          sleep 2

          cd /path/to/LawLLM
          source .venv/bin/activate
          nohup streamlit run ui/app.py \
            --server.port 8501 \
            --server.address 0.0.0.0 \
            > logs/ui.log 2>&1 &

          sleep 5
          curl -sf http://localhost:8501 || exit 1

      - name: Notify deployment
        if: success()
        run: echo "✅ 배포 완료 - $(date)"

      - name: Notify failure
        if: failure()
        run: echo "❌ 배포 실패 - $(date)"
```

### 7.2 환경 변수 관리

```bash
# .env (git 무시)
CONFIG_PATH=config.prod.yaml
```

## 8. 릴리스 전략

### 8.1 버전 관리 (Semantic Versioning)

```
v{MAJOR}.{MINOR}.{PATCH}

MAJOR: 호환성 깨지는 변경 (API 스키마 변경, 모델 교체)
MINOR: 기능 추가 (새 엔드포인트, RAG 개선)
PATCH: 버그 수정, 데이터 추가

예시:
v0.1.0  초기 릴리스 (MVP)
v0.2.0  스트리밍 응답 추가
v0.2.1  빈 검색 결과 에러 수정
v0.3.0  임대차 판례 5,000건 추가
v1.0.0  첫 정식 릴리스
```

### 8.2 릴리스 워크플로

```bash
# 1. main에서 릴리스 태그 생성
git tag -a v0.2.0 -m "feat: 스트리밍 응답 및 피드백 기능 추가"
git push origin v0.2.0

# 2. GitHub Release 생성
gh release create v0.2.0 \
  --title "v0.2.0 - 스트리밍 응답" \
  --notes "## 변경 사항
- SSE 기반 스트리밍 응답 추가
- 사용자 피드백 수집 API 추가
- 임대차 관련 판례 2,000건 추가"
```

### 8.3 릴리스 체크리스트

```
릴리스 전:
  □ main 브랜치 최신 상태
  □ 모든 CI 테스트 통과
  □ 로컬 M4에서 통합 테스트 통과
  □ E2E 수동 테스트 통과
  □ config.yaml 버전 업데이트
  □ CHANGELOG 작성

릴리스 후:
  □ 배포 서버 헬스체크 정상
  □ 채팅 테스트 1회 이상 수행
  □ GitHub Release 노트 작성
```

## 9. 모니터링 및 롤백

### 9.1 배포 후 모니터링

```bash
# 헬스체크 (1분 간격 자동)
# 이 스크립트를 crontab에 등록
#!/bin/bash
# scripts/healthcheck.sh

API_OK=$(curl -sf http://localhost:8000/api/health | python3 -c "
import sys, json
data = json.load(sys.stdin)
print('ok' if data['status'] == 'ok' and data['model_loaded'] else 'fail')
" 2>/dev/null)

if [ "$API_OK" != "ok" ]; then
    echo "[$(date)] API 이상 감지" >> logs/alert.log
    # 선택: 자동 재시작 또는 알림 전송
fi
```

### 9.2 롤백 절차

```bash
# 방법 1: Git 태그로 롤백
git checkout v0.1.0
source .venv/bin/activate
pip install -r requirements.txt
# 서비스 재시작

# 방법 2: Git revert (이력 보존)
git revert HEAD
git push origin main
# CD가 자동으로 이전 상태로 배포

# 방법 3: Docker (이미지 태그 기반)
docker compose down
docker compose up -d legal-llm:v0.1.0
```
