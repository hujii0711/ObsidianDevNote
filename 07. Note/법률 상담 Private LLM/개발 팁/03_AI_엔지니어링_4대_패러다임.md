# AI 엔지니어링 4대 패러다임

> 프롬프트 엔지니어링 → 컨텍스트 엔지니어링 → 하네스 엔지니어링 → 에이전틱 엔지니어링
> 진화하는 AI 활용 방법론의 전체 그림

---

## 1. 한눈에 보는 4대 패러다임

```
                        자율성 높음
                            ▲
                            │
              ┌─────────────────────────────┐
              │   에이전틱 엔지니어링         │  AI가 스스로 판단하고
              │   (Agentic Engineering)      │  도구를 선택하여 실행
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │   하네스 엔지니어링           │  AI의 전체 동작 환경을
              │   (Harness Engineering)      │  설계하고 제약
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │   컨텍스트 엔지니어링         │  AI에게 주어지는 정보의
              │   (Context Engineering)      │  구조와 품질을 설계
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │   프롬프트 엔지니어링         │  AI에게 보내는 한 번의
              │   (Prompt Engineering)       │  질문/지시를 최적화
              └─────────────────────────────┘
                            │
                            ▼
                       자율성 낮음
```

## 2. 각 패러다임 개요

---

### 2.1 프롬프트 엔지니어링 (Prompt Engineering)

#### 정의

```
AI에게 보내는 하나의 질문 또는 지시문을 최적화하여
원하는 품질의 응답을 이끌어내는 기술
```

#### 핵심 질문

> "이 한 번의 입력을 어떻게 써야 최고의 출력을 얻을 수 있는가?"

#### 범위

```
┌──────────────────────────────┐
│         하나의 프롬프트        │
│                              │
│  시스템 지시 + 사용자 질문     │
│  + 예시 + 출력 형식 지정      │
│                              │
└──────────────────────────────┘
```

#### 핵심 기법

| 기법 | 설명 | 예시 |
|------|------|------|
| **Zero-shot** | 예시 없이 지시만 | "민사소송 절차를 설명해줘" |
| **Few-shot** | 예시를 포함 | "Q: ~~ A: ~~ 형식으로 답해줘" |
| **Chain-of-Thought** | 단계적 추론 유도 | "단계별로 생각해서 답해줘" |
| **Role Playing** | 역할 부여 | "너는 민사 전문 변호사야" |
| **Output Format** | 출력 형식 지정 | "JSON으로 답해줘" |
| **Constraint** | 제약 조건 명시 | "200자 이내로", "추측하지 마" |

#### 한계

```
- 매번 같은 지시를 반복해야 함 (상태 없음)
- 긴 대화에서 초기 지시가 희석됨
- 복잡한 작업을 하나의 프롬프트로 해결하기 어려움
- 외부 데이터를 활용할 수 없음
```

#### 비유

```
프롬프트 엔지니어링 = 한 장의 메모를 잘 쓰는 기술
```

---

### 2.2 컨텍스트 엔지니어링 (Context Engineering)

#### 정의

```
AI에게 제공되는 모든 정보(컨텍스트)의 구조, 품질, 순서, 양을
체계적으로 설계하여 AI의 이해와 판단의 질을 높이는 기술
```

#### 핵심 질문

> "AI가 최적의 판단을 내리려면 어떤 정보를, 어떤 구조로, 얼마나 제공해야 하는가?"

#### 범위

```
┌───────────────────────────────────────────────────┐
│                  컨텍스트 윈도우                     │
│                                                    │
│  ┌────────────┐  ┌──────────────┐  ┌───────────┐ │
│  │ 시스템      │  │ 검색된 문서   │  │ 대화 이력  │ │
│  │ 프롬프트    │  │ (RAG 결과)   │  │           │ │
│  └────────────┘  └──────────────┘  └───────────┘ │
│  ┌────────────┐  ┌──────────────┐  ┌───────────┐ │
│  │ 코드베이스  │  │ 설정 파일    │  │ 사용자     │ │
│  │ 정보       │  │             │  │ 프로필     │ │
│  └────────────┘  └──────────────┘  └───────────┘ │
│                                                    │
│  → 이 모든 것의 선택, 구조, 순서, 압축을 설계       │
└───────────────────────────────────────────────────┘
```

#### 핵심 기법

| 기법 | 설명 | 이 프로젝트 적용 |
|------|------|-----------------|
| **RAG** | 외부 문서를 검색하여 컨텍스트에 삽입 | 법조문/판례 검색 → 프롬프트에 포함 |
| **컨텍스트 선택** | 관련 있는 정보만 선별 | 리랭킹으로 top-3만 전달 |
| **컨텍스트 압축** | 긴 문서를 요약하여 전달 | 판례 전문 → 판시사항만 추출 |
| **컨텍스트 순서** | 중요한 정보를 앞/뒤에 배치 | 시스템 프롬프트 → RAG 결과 → 질문 순서 |
| **메모리** | 이전 대화/세션 정보 관리 | 대화 이력 DB, Claude Memory |
| **구조화** | 정보를 명확한 구조로 전달 | YAML/JSON 형식, 섹션 구분 |

#### 프롬프트 엔지니어링과의 차이

```
프롬프트 엔지니어링:  "지시를 잘 쓰기"    → 어떻게 물어볼까?
컨텍스트 엔지니어링:  "참고 자료를 잘 주기" → 무엇을 알려줄까?

프롬프트:  "임대차 보증금 반환 절차를 알려줘"
컨텍스트:  "참고: 주택임대차보호법 제3조의3에 따르면... [검색 결과]"
           "이전 대화에서 사용자는 전세 계약 상황을 설명했음"
           "사용자는 법률 비전문가임"

둘 다 합쳐져야 좋은 응답이 나옴
```

#### 비유

```
프롬프트 엔지니어링 = 질문을 잘 쓰는 기술
컨텍스트 엔지니어링 = 질문과 함께 필요한 서류를 잘 준비하는 기술
```

### 2.3 하네스 엔지니어링 (Harness Engineering)

#### 정의

```
AI 에이전트가 동작하는 전체 환경(하네스)을 설계하여
에이전트의 행동 범위, 규칙, 도구, 자동화를 체계적으로 구성하는 기술
```

#### 핵심 질문

> "AI가 이 프로젝트에서 일관되게 올바른 행동을 하려면
>  어떤 규칙, 도구, 자동화가 필요한가?"

#### 범위

```
┌──────────────────────────────────────────────────────────┐
│                        하네스                             │
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │  CLAUDE.md   │  │  settings    │  │  MCP 서버     │  │
│  │  (규칙)      │  │  (권한/훅)   │  │  (외부 연동)  │  │
│  └──────────────┘  └──────────────┘  └───────────────┘  │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │  Skills      │  │  Agents      │  │  docs/        │  │
│  │  (명령)      │  │  (전문가)    │  │  (참조 문서)  │  │
│  └──────────────┘  └──────────────┘  └───────────────┘  │
│                                                          │
│  → 이 전체 환경이 AI의 행동을 결정                        │
└──────────────────────────────────────────────────────────┘
```

#### 핵심 기법

| 기법 | 설명 | 이 프로젝트 적용 |
|------|------|-----------------|
| **규칙 정의** | 프로젝트 규칙을 CLAUDE.md에 명시 | "타입 힌트 필수", "async 필수" |
| **권한 관리** | 허용/금지 명령 설정 | "pytest 허용", "rm -rf 금지" |
| **이벤트 훅** | 특정 이벤트에 자동 실행 | "파일 수정 후 자동 린트" |
| **도구 연결** | MCP로 외부 시스템 접근 | GitHub, SQLite, Fetch |
| **작업 위임** | 서브 에이전트에게 전문 작업 위임 | 코드 리뷰, 데이터 검증 |
| **계층화** | 규칙을 계층적으로 구성 | 글로벌 → 프로젝트 → 디렉터리 |
| **반복 자동화** | 자주 하는 작업을 스킬로 정의 | /test, /lint, /deploy-check |

#### 컨텍스트 엔지니어링과의 차이

```
컨텍스트 엔지니어링:  "이번 대화에서 무엇을 알려줄까?"  → 정보 설계
하네스 엔지니어링:    "모든 대화에서 어떻게 행동할까?"   → 환경 설계

컨텍스트:  이번 질문에 필요한 법조문 3개를 검색해서 전달
하네스:    어떤 질문이든 면책 조항을 포함하도록 규칙 설정
           파일 수정 시 항상 자동 린트되도록 훅 설정
           GitHub 이슈를 확인할 수 있도록 MCP 연결

컨텍스트는 대화마다 달라지고,
하네스는 대화와 무관하게 항상 적용됨
```

#### 비유

```
프롬프트 엔지니어링 = 질문을 잘 쓰는 기술
컨텍스트 엔지니어링 = 참고 서류를 잘 준비하는 기술
하네스 엔지니어링   = 사무실 환경 자체를 설계하는 기술
                     (업무 규칙, 도구 배치, 자동화 시스템)
```

### 2.4 에이전틱 엔지니어링 (Agentic Engineering)

#### 정의

```
AI가 단순히 질문에 답하는 것을 넘어, 스스로 계획을 세우고
도구를 선택하며, 결과를 평가하고, 필요시 재시도하는
자율적 에이전트 시스템을 설계하는 기술
```

#### 핵심 질문

> "AI가 스스로 판단하고 행동하는 자율적 시스템을
>  어떻게 안전하고 효과적으로 설계할 수 있는가?"

#### 범위

```
┌──────────────────────────────────────────────────────────┐
│                    에이전틱 시스템                         │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │  인지     │  │  계획     │  │  실행     │  │  평가   │ │
│  │ Perceive │  │  Plan    │  │  Act     │  │ Evaluate│ │
│  │          │  │          │  │          │  │         │ │
│  │ 상황 파악 │  │ 단계 설계 │  │ 도구 호출 │  │ 결과 검증│ │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬────┘ │
│       └──────────────┴──────────────┴──────────────┘     │
│                         ↻ 반복                           │
│                                                          │
│  ┌──────────────────────────────────────────────────┐    │
│  │  멀티 에이전트 협업                                │    │
│  │  ┌────────┐  ┌────────┐  ┌────────┐             │    │
│  │  │ Agent A│←→│ Agent B│←→│ Agent C│             │    │
│  │  │(코드)  │  │(리뷰)  │  │(테스트) │             │    │
│  │  └────────┘  └────────┘  └────────┘             │    │
│  └──────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘
```

#### 핵심 기법

| 기법 | 설명 | Claude Code 대응 |
|------|------|-----------------|
| **계획 수립** | 작업을 단계로 분해 | Plan 모드, TaskCreate |
| **도구 사용** | 적절한 도구를 선택 실행 | Bash, Read, Edit, Grep, MCP |
| **자기 평가** | 결과를 스스로 검증 | 테스트 실행 후 결과 분석 |
| **재시도** | 실패 시 다른 접근 시도 | 에러 분석 → 수정 → 재실행 |
| **위임** | 전문 작업을 서브 에이전트에 할당 | Agent 도구로 서브 에이전트 호출 |
| **메모리** | 이전 결과를 기억하고 활용 | Claude Memory, 파일 기반 메모리 |
| **자율 루프** | 목표 달성까지 반복 | /loop 스킬, 자율 에이전트 |

#### 에이전틱 동작의 실제 예시

```
사용자: "GitHub 이슈 #15의 버그를 수정해줘"

에이전틱 동작 (Claude Code가 자율적으로):
  
  [인지] GitHub MCP로 이슈 #15 확인
         → "RAG 검색 결과 빈 배열 시 500 에러"
  
  [계획] 수정 계획 수립
         1. routes.py에서 빈 배열 처리 로직 확인
         2. 에러 재현 조건 파악
         3. 수정 코드 작성
         4. 테스트 추가
         5. PR 생성
  
  [실행] 
         Read routes.py → 빈 배열 처리 누락 발견
         Edit routes.py → 빈 배열 시 기본 응답 처리 추가
         Write test_api.py → test_chat_empty_rag_results 추가
  
  [평가]
         pytest tests/test_api.py → 1 failed
         → "fixture에 mock_retriever 누락"
  
  [재시도]
         Edit conftest.py → mock에 빈 배열 반환 케이스 추가
         pytest tests/test_api.py → all passed ✅
  
  [완료]
         git commit → git push → gh pr create
         PR에 이슈 #15 링크
```

#### 하네스 엔지니어링과의 차이

```
하네스 엔지니어링:    "AI의 행동 환경을 설계" → 규칙과 도구를 제공
에이전틱 엔지니어링:  "AI가 자율적으로 행동"  → 그 환경 안에서 스스로 판단

하네스:    "pytest는 허용, rm -rf는 금지" (환경 규칙)
에이전틱:  "테스트가 실패하면 원인을 분석하고 수정 후 재실행" (자율 행동)

하네스는 '울타리'를 치는 것이고,
에이전틱은 그 울타리 안에서 AI가 '스스로 뛰어다니는' 것
```

#### 비유

```
프롬프트 엔지니어링 = 질문을 잘 쓰는 기술
컨텍스트 엔지니어링 = 참고 서류를 잘 준비하는 기술
하네스 엔지니어링   = 사무실 환경을 설계하는 기술
에이전틱 엔지니어링 = 자율적으로 일하는 직원을 육성하는 기술
                     (판단력, 도구 활용, 협업, 자기 검증)
```

## 3. 4대 패러다임 상세 비교

### 3.1 핵심 차원 비교

| 차원 | 프롬프트 | 컨텍스트 | 하네스 | 에이전틱 |
|------|---------|---------|--------|---------|
| **설계 대상** | 하나의 입력 텍스트 | 모든 입력 정보 | 동작 환경 전체 | 자율 행동 시스템 |
| **지속성** | 일회성 | 세션/작업 단위 | 프로젝트 영구 | 프로젝트 영구 |
| **상태** | 무상태 | 상태 있음 (메모리) | 상태 있음 (설정) | 상태 있음 (계획+메모리) |
| **AI 자율성** | 없음 (수동 응답) | 낮음 (정보 활용) | 중간 (규칙 준수) | 높음 (자율 판단) |
| **복잡도** | 낮음 | 중간 | 높음 | 매우 높음 |
| **실패 모드** | 나쁜 응답 | 잘못된 정보 활용 | 규칙 미준수 | 의도치 않은 행동 |

### 3.2 입력-출력 비교

```
프롬프트 엔지니어링:
  입력: "보증금 반환 절차 알려줘"
  출력: "보증금 반환 절차는..."
  ────────────────────────────
  1번의 질문 → 1번의 답변

컨텍스트 엔지니어링:
  입력: 시스템 프롬프트 + RAG 검색 결과 3건 + 대화 이력 + 질문
  출력: 출처가 명시된 구조화된 답변
  ────────────────────────────
  풍부한 정보 → 정확한 답변

하네스 엔지니어링:
  입력: CLAUDE.md 규칙 + settings 권한 + MCP 연결 + 질문
  출력: 규칙을 준수하는 코드 + 자동 린트 + 자동 테스트
  ────────────────────────────
  환경 설계 → 일관된 행동

에이전틱 엔지니어링:
  입력: "이슈 #15 수정해줘"
  출력: 이슈 분석 → 코드 수정 → 테스트 → PR 생성 (자율 수행)
  ────────────────────────────
  목표 제시 → 자율적 문제 해결
```

  │
  ▼

## 4. 이 프로젝트에서의 통합 적용

### 4.1 각 계층이 담당하는 영역

```
사용자: "임대차 관련 판례 데이터 추가하고 RAG 품질 테스트해줘"

[프롬프트 엔지니어링 계층]
  └── build_chat_prompt()에서 "참고 자료에 근거하여 답변하세요" 지시

[컨텍스트 엔지니어링 계층]
  ├── RAG가 관련 법조문/판례를 검색하여 컨텍스트에 삽입
  ├── 리랭킹으로 가장 관련 높은 3건만 선택
  └── 대화 이력을 DB에서 조회하여 맥락 유지

[하네스 엔지니어링 계층]
  ├── CLAUDE.md: "데이터 원본은 수정하지 않음", "개인정보 비식별화 필수"
  ├── afterEdit 훅: 코드 수정 시 자동 린트
  ├── /rag-test 스킬: RAG 품질 테스트 절차 정의
  └── data-validator 서브 에이전트: 데이터 품질 자동 검증

[에이전틱 엔지니어링 계층]
  ├── [인지] "판례 데이터 추가" + "RAG 품질 테스트" 두 작업 파악
  ├── [계획] 1. 데이터 검증 → 2. 저장 → 3. 인덱싱 → 4. 품질 테스트
  ├── [실행] 각 단계를 도구/에이전트로 실행
  ├── [평가] 테스트 결과 분석, 검색 정확도 확인
  └── [재시도] 품질 미달 시 청킹 파라미터 조정 후 재인덱싱
```

### 4.2 계층 간 의존 관계

```
에이전틱이 잘 동작하려면:
  └── 하네스가 잘 설계되어야 함 (규칙, 도구, 권한)
       └── 컨텍스트가 잘 구성되어야 함 (RAG, 메모리)
            └── 프롬프트가 잘 작성되어야 함 (지시, 제약)

역으로:
  프롬프트가 아무리 좋아도 → 컨텍스트(RAG 결과)가 없으면 환각
  컨텍스트가 풍부해도 → 하네스(규칙)가 없으면 일관성 없는 코드
  하네스가 잘 설계되어도 → 에이전틱(자율성)이 없으면 매번 수동 지시
```

## 5. 성숙도 모델

### 5.1 AI 활용 성숙도 5단계

```
Level 0: 수동 코딩
  AI 사용하지 않음. 개발자가 모든 것을 직접 작성.

Level 1: 프롬프트 보조
  ChatGPT/Copilot에 질문하여 코드 조각을 얻음.
  "이 함수 작성해줘" → 복사-붙여넣기
  ──── 프롬프트 엔지니어링 ────

Level 2: 컨텍스트 보조
  RAG, 코드베이스 인덱싱으로 AI에게 풍부한 정보 제공.
  AI가 프로젝트 맥락을 이해하고 더 정확한 코드 생성.
  ──── 컨텍스트 엔지니어링 ────

Level 3: 하네스 기반 개발
  CLAUDE.md, 스킬, 훅으로 AI의 행동을 체계적으로 관리.
  규칙 위반 자동 방지, 반복 작업 자동화.
  ──── 하네스 엔지니어링 ────

Level 4: AI 주도 개발
  AI가 계획 수립, 코드 작성, 테스트, PR 생성을 자율 수행.
  개발자는 의사결정과 검증에 집중.
  ──── 에이전틱 엔지니어링 ────

Level 5: 자율 에이전트
  AI가 이슈 감지, 수정, 배포까지 독립적으로 수행.
  개발자는 감독과 방향 설정만.
  ──── 에이전틱 엔지니어링 (고도화) ────
```

### 5.2 이 프로젝트의 목표 성숙도

```
현재 설계: Level 4 (AI 주도 개발)

구성:
  Level 1 ✅ 프롬프트: prompt.py 템플릿 최적화
  Level 2 ✅ 컨텍스트: RAG + 리랭킹 + 대화 이력
  Level 3 ✅ 하네스: CLAUDE.md + settings + hooks + skills + agents
  Level 4 ✅ 에이전틱: 자율 계획-실행-평가, 서브 에이전트 협업
```

## 6. 안티패턴과 함정

### 6.1 각 계층의 흔한 실수

| 계층 | 안티패턴 | 올바른 접근 |
|------|---------|-----------|
| **프롬프트** | 너무 길고 복잡한 지시 | 핵심만 간결하게, 복잡한 건 컨텍스트로 |
| **프롬프트** | 모호한 지시 ("잘 해줘") | 구체적 기준 제시 ("200자 이내, JSON 형식") |
| **컨텍스트** | 관련 없는 정보까지 모두 전달 | 리랭킹으로 top-k만 선별 |
| **컨텍스트** | 컨텍스트 윈도우 초과 | 압축, 요약, 선택적 로딩 |
| **하네스** | CLAUDE.md에 모든 것을 넣음 | 200줄 이내 + docs/ 분리 |
| **하네스** | 너무 많은 규칙 | 핵심 규칙만, "3회 반복 시 규칙화" |
| **에이전틱** | 위험한 명령에 대한 자율성 | permissions deny로 안전장치 |
| **에이전틱** | 무한 루프 (재시도 반복) | 최대 재시도 횟수 설정 |

### 6.2 계층 간 불균형

```
❌ 프롬프트만 잘 짜고 나머지 없음
  → 매번 같은 지시 반복, 프로젝트 규칙 누락

❌ 하네스만 복잡하고 프롬프트/컨텍스트 부실
  → 규칙은 많은데 AI가 참조할 정보가 없음

❌ 에이전틱에 의존하고 하네스 없음
  → AI가 자율적으로 움직이지만 방향이 없음 (위험)

✅ 4계층이 균형 있게 설계
  → 좋은 프롬프트 + 풍부한 컨텍스트 + 일관된 하네스 + 자율적 에이전트
```
