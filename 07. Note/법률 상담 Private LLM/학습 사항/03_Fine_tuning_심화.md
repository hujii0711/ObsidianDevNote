> LoRA/QLoRA의 원리를 이해하고, 법률 도메인 파인튜닝을 올바르게 수행하기 위한 가이드

## 1. Fine-tuning의 종류

### 전체 그림

```
Pre-training (사전학습)
  │  수조 토큰으로 언어 자체를 학습 (수백만 달러)
  │  → 우리가 할 일이 아님
  ▼
Fine-tuning (미세조정)  ← 이 프로젝트에서 수행
  │  도메인 데이터로 특화 학습 (수시간~수일)
  ▼
Inference (추론)
     학습된 모델로 응답 생성
```

### Fine-tuning 방법 비교

| 방법 | 수정 범위 | 메모리 | 성능 | 이 프로젝트 |
|------|----------|--------|------|-----------|
| Full Fine-tuning | 전체 파라미터 | 매우 높음 | 최고 | 불가 (메모리 부족) |
| **LoRA** | 소규모 어댑터만 | 낮음 | 우수 | 가능 |
| **QLoRA** | 양자화 + LoRA | **매우 낮음** | 우수 | **사용 (권장)** |
| Prefix Tuning | 프리픽스 벡터만 | 매우 낮음 | 보통 | 미사용 |
| Adapter Tuning | 어댑터 레이어 삽입 | 낮음 | 우수 | 미사용 |

## 2. LoRA (Low-Rank Adaptation) 원리

### 2.1 핵심 아이디어

```
기존 Full Fine-tuning:
  W_new = W_old + ΔW        (ΔW는 W와 같은 크기 → 메모리 큼)

LoRA:
  W_new = W_old + A × B     (A: d×r, B: r×d, r << d)
  
  W_old: 원본 가중치 (고정, 수정하지 않음)
  A × B: 저랭크(low-rank) 근사 행렬 (학습 대상)
  r: 랭크 (예: 64) ← 이것만 학습
```

### 2.2 시각적 이해

```
원본 가중치 W (10000 × 10000 = 1억 개 파라미터)
  → Full Fine-tuning: 1억 개 전부 업데이트

LoRA (rank=64):
  A: 10000 × 64 = 64만 개
  B: 64 × 10000 = 64만 개
  → 총 128만 개만 학습 (원본의 1.28%)

메모리 절약: ~98% 감소!
성능: Full Fine-tuning의 ~95-99% 수준
```

### 2.3 LoRA 핵심 파라미터

| 파라미터 | 의미 | 이 프로젝트 설정 |
|---------|------|----------------|
| **rank (r)** | 저랭크 행렬의 차원, 높을수록 표현력 증가 | 64 |
| **alpha (α)** | 스케일링 팩터, LoRA 가중치의 영향력 | 128 |
| **target_modules** | LoRA를 적용할 레이어 | q_proj, k_proj, v_proj, o_proj |
| **lora_layers** | LoRA를 적용할 레이어 수 | 16 |

### 2.4 rank와 alpha의 관계

```
실제 적용 비율 = alpha / rank = 128 / 64 = 2.0

rank를 높이면:
  → 더 많은 정보를 학습할 수 있음
  → 메모리 사용량 증가
  → 과적합 위험 증가

alpha를 높이면:
  → LoRA의 영향력 증가
  → 원본 모델 지식 덜 보존

일반 규칙: alpha = rank × 2 (이 프로젝트도 이 규칙 따름)
```

## 3. QLoRA 특수 사항

### 3.1 QLoRA = 양자화 + LoRA

```
일반 LoRA:
  베이스 모델 (FP16, ~22GB) + LoRA 어댑터 학습
  → 총 메모리 ~30GB 필요

QLoRA:
  베이스 모델 (Q4, ~8GB) + LoRA 어댑터 학습  ← 이 프로젝트
  → 총 메모리 ~16GB 필요
  → M4 48GB에서 넉넉하게 학습 가능
```

### 3.2 NF4 (Normal Float 4-bit)

```
QLoRA에서 사용하는 특수 4비트 양자화 방식:
- 가중치 분포가 정규분포를 따른다는 점을 활용
- 일반 INT4보다 정보 손실이 적음
- Double Quantization으로 양자화 상수도 양자화
```

## 4. Instruction Tuning 데이터 설계

### 4.1 데이터 품질 > 데이터 양

```
나쁜 데이터 10,000건 < 좋은 데이터 1,000건

좋은 학습 데이터의 조건:
1. 질문이 구체적이고 명확함
2. 답변이 정확하고 완결됨
3. 답변 형식이 일관됨 (단계별 안내, 법조문 인용 등)
4. 다양한 민사 분야를 골고루 포함
```

### 4.2 답변 형식 표준화

```json
{
  "instruction": "전세 보증금을 돌려받지 못하면 어떻게 해야 하나요?",
  "input": "",
  "output": "전세 보증금 미반환 시 다음 절차를 진행할 수 있습니다.\n\n1. **내용증명 발송**: ...\n2. **임차권등기명령**: ...(주택임대차보호법 제3조의3)\n3. **지급명령 신청**: ...\n4. **민사소송 제기**: ...\n\n참고로, 소액사건(3,000만원 이하)의 경우..."
}
```

**일관되어야 할 요소:**
- 서두에 핵심 답변 제시
- 번호 매긴 단계별 설명
- 관련 법조문 괄호 표기
- 추가 참고사항 안내

### 4.3 카테고리별 균형

```
이상적인 카테고리 분포:
  임대차/부동산: 25%
  손해배상:     20%
  계약/매매:    15%
  채권/추심:    15%
  가사(이혼):   10%
  상속:         10%
  기타:          5%
```

## 5. 학습 과정 이해

### 5.1 Loss 곡선 해석

```
이상적인 곡선:
Loss │
  3  │╲
     │ ╲
  2  │  ╲
     │   ╲___
  1  │       ╲____
     │             ╲________  ← 수렴
  0  └───────────────────── Iterations

경고 신호:
- Loss가 전혀 줄지 않음 → learning_rate 너무 낮음
- Loss가 급격히 튐 → learning_rate 너무 높음
- Loss가 0에 수렴 → 과적합
- Loss가 특정 값에서 정체 → 데이터 품질 또는 rank 부족
```

### 5.2 과적합 vs 과소적합

```
과적합 (Overfitting):
  - 학습 데이터는 잘 맞추지만 새 질문에 대해 잘못된 답변
  - 학습 데이터를 거의 외워버림
  - 해결: epoch 줄이기, rank 낮추기, 데이터 증강

과소적합 (Underfitting):
  - 학습 데이터조차 잘 못 맞춤
  - 학습이 충분하지 않음
  - 해결: epoch 늘리기, rank 높이기, learning_rate 조정
```

### 5.3 Gradient Accumulation 이해

```
목표 배치 크기: 32
실제 배치 크기: 4 (메모리 제한)
Gradient Accumulation: 8

→ 4 × 8 = 32 (실제 효과적 배치 크기)
→ 8번의 미니배치 기울기를 누적한 후 한 번 업데이트
→ 메모리는 배치 4만큼만 사용하면서 배치 32의 효과
```

## 6. 학습 결과 평가 방법

### 6.1 자동 평가 지표

| 지표 | 설명 | 용도 |
|------|------|------|
| **Perplexity** | 모델의 예측 불확실성 (낮을수록 좋음) | 기본 언어 모델 품질 |
| **BLEU** | 참조 답변과의 n-gram 일치도 | 번역/생성 품질 |
| **ROUGE** | 참조 답변과의 텍스트 겹침 | 요약 품질 |
| **BERTScore** | 의미적 유사도 (임베딩 기반) | 의미 보존 |

### 6.2 사람 평가 (이 프로젝트에서 가장 중요)

```
평가 기준표:

정확성 (1-5):  법률적으로 정확한 정보를 제공하는가?
완결성 (1-5):  필요한 모든 정보를 포함하는가?
안전성 (1-5):  위험하거나 잘못된 조언이 없는가?
자연스러움 (1-5): 읽기 쉽고 이해하기 쉬운가?

→ 테스트 세트 50건에 대해 Before(베이스)/After(파인튜닝) 비교
```
