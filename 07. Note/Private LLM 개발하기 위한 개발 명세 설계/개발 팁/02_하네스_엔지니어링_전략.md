# 10. 하네스 엔지니어링 전략

> Claude Code 하네스(harness)의 모든 설정 요소를 최적 설계하여 AI가 프로젝트 규칙을 정확히 이해하고 일관되게 동작하도록 만드는 전략

---

## 1. 하네스 엔지니어링이란?

### 1.1 정의

```
프롬프트 엔지니어링:  하나의 질문에 대한 최적의 프롬프트를 설계
하네스 엔지니어링:    AI 에이전트가 동작하는 전체 환경을 설계

하네스 = CLAUDE.md + settings.json + MCP + Skills + Hooks + Agents + docs/
        ───────────────────────────────────────────────────────────────
        이 모든 것이 Claude Code가 "어떻게 행동할지"를 결정하는 하네스
```

### 1.2 왜 중요한가?

```
하네스 없이:
  매번 "타입 힌트 꼭 써줘", "async로 작성해줘", "하드코딩하지 마"
  → 반복 지시, 누락, 일관성 없는 코드

하네스 설계 후:
  한 번 설정 → 모든 대화에서 규칙 자동 적용
  → 지시 없이도 일관된 코드 품질 유지
```

### 1.3 하네스 구성 요소 전체 맵

```
~/.claude/                          ← 글로벌 (모든 프로젝트)
├── settings.json                   ← 사용자 설정
└── CLAUDE.md                       ← 개인 작업 스타일

프로젝트 루트/
├── .claude/
│   ├── settings.json               ← 프로젝트 설정 (공유)
│   ├── settings.local.json         ← 로컬 설정 (비공유)
│   ├── CLAUDE.md                   ← 프로젝트 규칙 (핵심)
│   ├── docs/                       ← 참조 문서
│   │   ├── API_SCHEMA.md
│   │   ├── DATA_FORMAT.md
│   │   └── TESTING_POLICY.md
│   ├── commands/                   ← 커스텀 스킬
│   │   ├── test.md
│   │   └── deploy-check.md
│   └── agents/                     ← 서브 에이전트
│       ├── code-reviewer.md
│       └── data-validator.md
├── .mcp.json                       ← MCP 서버 연결
├── CLAUDE.md                       ← 프로젝트 최상위 규칙
└── 하위 디렉터리/CLAUDE.md          ← 디렉터리별 규칙
```

## 2. CLAUDE.md 계층 설계

### 2.1 로딩 순서와 우선순위

```
Claude Code가 대화를 시작하면 다음 순서로 CLAUDE.md를 로딩:

1. ~/.claude/CLAUDE.md                    ← 개인 글로벌 (가장 낮은 우선순위)
2. 프로젝트루트/CLAUDE.md                  ← 프로젝트 공통
3. 프로젝트루트/.claude/CLAUDE.md          ← 프로젝트 상세
4. 작업 중인 디렉터리/CLAUDE.md            ← 디렉터리 특화 (가장 높은 우선순위)

하위가 상위를 오버라이드하지는 않지만, 더 구체적인 규칙이 우선 적용됩니다.
```

### 2.2 계층별 역할 설계

#### Layer 1: 글로벌 (~/.claude/CLAUDE.md)

```markdown

- 불확실한 사항은 먼저 질문하세요.

## 코딩 스타일
- 모든 코드에 타입 힌트를 포함하세요.
- 한국어 주석은 함수 docstring에만 사용하세요.

...

...

#### Layer 3: 디렉터리별 (src/rag/CLAUDE.md)

```markdown
# RAG 파이프라인 규칙

## 이 디렉터리의 책임
- 문서 인덱싱, 벡터 검색, 리랭킹
- 임베딩 모델은 이 디렉터리 내에서만 로드

## 모듈 구조
- indexer.py: 문서 → ChromaDB 인덱싱 (CLI 도구)
- retriever.py: 벡터 검색 + 리랭킹 통합 (싱글턴)
- reranker.py: KoSentenceBERT CrossEncoder

## 수정 시 주의사항
- retriever.py의 Retriever는 싱글턴. 생성자 변경 시 main.py lifespan 확인
- 청킹 파라미터는 config.yaml에서 관리. 이 파일에 하드코딩 금지
- 새 검색 전략 추가 시 retriever.py의 search() 인터페이스 유지
```

## 3. CLAUDE.md 작성 기법

### 3.1 규칙 작성의 원칙

```
❌ 모호한 규칙:
  "코드를 깔끔하게 작성하세요."
  "적절한 에러 처리를 하세요."

✅ 구체적인 규칙:
  "모든 함수에 타입 힌트를 포함하세요. Any 사용 금지."
  "외부 경계(API 입력, 파일 I/O, 모델 로딩)에서만 try/except를 사용하세요."
```

### 3.2 규칙의 5가지 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **금지 규칙** | 절대 하지 말 것 | "LangChain 사용 금지", "Any 타입 금지" |
| **필수 규칙** | 반드시 할 것 | "타입 힌트 필수", "면책 조항 필수" |
| **선호 규칙** | 가능하면 따를 것 | "가능하면 컴프리헨션 사용" |
| **조건부 규칙** | 특정 상황에서 적용 | "FastAPI 핸들러는 반드시 async def" |
| **참조 규칙** | 외부 문서 참조 | "API 스키마는 .claude/docs/API_SCHEMA.md 참조" |

### 3.3 규칙 간 충돌 방지

```markdown
<!-- 우선순위를 명시적으로 지정 -->
## 규칙 우선순위
1. 보안 규칙 (개인정보, 면책 조항) — 최우선
2. 아키텍처 규칙 (싱글턴, 비동기) — 높음
3. 코딩 스타일 (타입 힌트, 포매팅) — 중간
4. 성능 규칙 (메모리, 배치 크기) — 낮음

충돌 시 우선순위가 높은 규칙을 따르세요.
```

### 3.4 부정형보다 긍정형

```
❌ "하드코딩하지 마세요."
   → Claude가 "그러면 뭘 해야 하지?" 고민

✅ "모든 설정값은 config.yaml 또는 환경변수로 관리하세요."
   → 대안이 명확
```

```markdown

### 5.5 permissions 설계 전략

```
allow 설계 원칙:
  1. 자주 사용하는 명령만 허용 → 매번 승인 프롬프트 감소
  2. 위험한 명령은 deny에 명시 → 실수 방지
  3. 와일드카드(*) 범위를 최소화 → 보안

레벨별 접근:
  안전 (자동 허용):     읽기 전용 명령 (git status, pip list, curl localhost)
  보통 (자동 허용):     개발 도구 (ruff, pytest, uvicorn)
  주의 (승인 필요):     파일 삭제, git push, docker 실행
  금지 (차단):          rm -rf, force push, 파이프 실행 (curl|bash)
```

## 6. 훅 상세 설계

### 6.1 사용 가능한 훅 이벤트

| 이벤트 | 트리거 시점 | 활용 |
|--------|-----------|------|
| `afterEdit` | 파일 수정 후 | 자동 린트/포맷 |
| `preCommit` | 커밋 전 | 테스트 실행 |
| `onPromptSubmit` | 사용자 입력 시 | 컨텍스트 주입, 규칙 리마인드 |

### 6.2 afterEdit 훅 상세

```json
{
  "hooks": {
    "afterEdit": [
      {
        "matcher": "src/**/*.py",
        "command": "ruff check --fix $FILE 2>/dev/null; ruff format $FILE 2>/dev/null"
      },
      {
        "matcher": "tests/**/*.py",
        "command": "ruff check --fix $FILE 2>/dev/null; ruff format $FILE 2>/dev/null"
      },
      {
        "matcher": "**/*.yaml",
        "command": "python -c \"import yaml; yaml.safe_load(open('$FILE'))\" 2>&1 || echo 'YAML 구문 오류: $FILE'"
      }
    ]
  }
}
```

**동작 흐름:**
```
Claude가 src/rag/retriever.py 수정
    │
    ▼
[afterEdit 훅 실행]
  1. ruff check --fix retriever.py   ← 린트 위반 자동 수정
  2. ruff format retriever.py        ← 포맷 자동 적용
    │
    ▼
Claude가 수정된 최종 결과를 확인
```

### 6.3 preCommit 훅 상세

```json
{
  "hooks": {
    "preCommit": [
      {
        "command": "cd $PROJECT_DIR && pytest tests/ -x -q --tb=line 2>&1 | tail -20"
      }
    ]
  }
}
```

**동작 흐름:**
```
Claude가 git commit 시도
    │
    ▼
[preCommit 훅 실행]
  pytest tests/ -x -q
    │
  Pass ──→ 커밋 진행
  Fail ──→ 훅이 실패 메시지 반환
            │
            ▼
          Claude가 실패 원인 분석 → 코드 수정 → 재커밋
```

### 6.4 훅 디버깅

```
훅이 예상대로 동작하지 않을 때:

1. 명령을 터미널에서 직접 실행하여 확인
   $ ruff check --fix src/rag/retriever.py

2. $FILE, $PROJECT_DIR 변수가 올바르게 치환되는지 확인

3. 2>/dev/null 제거하여 에러 메시지 확인

4. 훅 명령의 exit code 확인
   - 0: 성공 (Claude에게 성공 보고)
   - 0이 아님: 실패 (Claude에게 실패 메시지 전달)
```

## 7. 커스텀 스킬 상세 설계

### 7.1 스킬 작성 원칙

```markdown
좋은 스킬의 특성:
  1. 명확한 단계: 번호 매긴 순차 지시
  2. 판단 기준 포함: "실패 시 ~하세요"
  3. 보고 형식 지정: "표로 정리", "체크리스트로"
  4. $ARGUMENTS 활용: 인자를 받아 유연하게 동작
  5. 안전장치: "원본은 수정하지 마세요"

나쁜 스킬:
  "테스트를 실행하세요." ← 너무 모호
  
좋은 스킬:
  "1. pytest tests/ -v를 실행하세요.
   2. 실패한 테스트가 있으면 원인을 분석하세요.
   3. 코드 수정이 필요하면 수정하고 재실행하세요.
   4. 결과를 ✅/❌ 표로 보고하세요." ← 구체적
```

### 7.2 $ARGUMENTS 활용 패턴

```markdown
<!-- 인자 없을 때 기본 동작, 있을 때 커스텀 동작 -->
인자가 있으면 해당 대상만 처리하세요: $ARGUMENTS
인자가 없으면 전체를 대상으로 실행하세요.

사용 예:
  /test                  → 전체 테스트 실행
  /test test_api.py      → test_api.py만 실행
  /test "rag"            → rag 관련 테스트만 실행
```

### 7.3 스킬 간 조합

```
사용자: /deploy-check

deploy-check 스킬 내부에서 다른 도구를 순차 호출:
  1. ruff check (린트)
  2. ruff format --check (포맷)
  3. pytest (테스트)
  4. git status (변경사항)
  5. config.yaml 검증
  6. 종합 보고
```

## 9. 하네스 최적화 기법

### 9.1 CLAUDE.md 길이 관리

```
문제: CLAUDE.md가 너무 길면 Claude의 주의력이 분산

  CLAUDE.md: 핵심 규칙만 (200줄 이내 권장)

분리 기준:
  매번 적용: CLAUDE.md에 유지
  특정 작업 시만: docs/로 분리
  특정 디렉터리만: 디렉터리 CLAUDE.md로 분리
```

### 9.2 규칙 효과 검증

```
새 규칙 추가 후 검증 방법:

1. 규칙을 의도적으로 위반하는 코드를 요청
   → Claude가 규칙에 따라 거부하거나 올바르게 수정하는지 확인

2. 규칙과 관련된 작업을 요청
   → 규칙이 자동으로 적용되는지 확인

3. 규칙을 언급하지 않고 작업 요청
   → 명시적 지시 없이도 규칙을 따르는지 확인

예시:
  규칙: "FastAPI 핸들러는 async def 필수"
  검증: "새 API 엔드포인트 추가해줘" → def가 아닌 async def로 작성하는지 확인
```
