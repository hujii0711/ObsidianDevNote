**Spec Driven Development(명세 기반 개발)** 현대 소프트웨어 공학에서 아주 중요한 **SDD**의 한 축입니다.
특히 API 설계나 복잡한 시스템을 구축할 때, 구현보다 '정의'를 앞세우는 이 방법론은 개발 효율성을 획기적으로 높여줍니다.

### Spec Driven Development (명세 기반 개발)이란?

단순히 문서를 만드는 것을 넘어, "명세서(Specification)가 개발의 중심이자 진실의 원천(Single Source of Truth)"이 되는 개발 방식입니다.

- **선 명세, 후 구현:** 코드를 짜기 전에 API의 입출력, 데이터 모델, 비즈니스 규칙을 담은 명세를 먼저 확정합니다.

- **자동화와의 연결:** 작성된 명세(Swagger/OpenAPI, Protobuf 등)를 바탕으로 클라이언트 SDK, 서버 스텁(Stub), 테스트 케이스, 더미 데이터를 자동으로 생성합니다.

- **커뮤니케이션 도구:** 프론트엔드와 백엔드 개발자가 서로의 구현이 끝나길 기다릴 필요 없이, 합의된 명세를 바탕으로 병렬 개발이 가능해집니다.

### 왜 요즘 더 주목받나요?

1. **API 퍼스트 설계:** MSA(마이크로서비스 아키텍처) 환경에서 서비스 간 통신 규약이 중요해지면서, 명세 기반 개발은 필수적인 요소가 되었습니다.

2. **LLM 및 AI 코딩 에이전트 활용:** 최근 **Cursor**나 **Claude Code** 같은 도구를 사용할 때, 명확한 **Spec**이 정의되어 있으면 AI가 훨씬 더 정확하고 일관성 있는 코드를 작성해 줍니다.

3. **LSP(Language Server Protocol) 개발:** VS Code 익스텐션 등을 개발할 때처럼 정해진 프로토콜 명세를 따라야 하는 경우, 이 방식이 가장 강력한 힘을 발휘합니다.

## gstack 개요

YC 대표 Garry Tan이 만든 오픈소스 프레임워크로, Claude Code를 "가상 엔지니어링 팀"으로 전환합니다. 23개의 전문가 역할(CEO, 디자이너, 엔지니어링 매니저, QA 리드 등)을 슬래시 커맨드(skills)로 구현했으며, MIT 라이선스로 무료 공개되어 있습니다.

핵심 철학은 **역할 분리**입니다. 단일 개발자가 gstack을 사용할 때 한 번에 하나의 일만 하는 게 아니라, 각 단계마다 정해진 담당자가 있는 파이프라인으로 작업을 이동시킵니다. 보안 리뷰를 어떻게 진행할지 고민할 필요 없이, 보안 리뷰 스킬이 이미 무엇을 해야 하는지 알고 있습니다.

주요 커맨드는 `/plan-ceo-review`(제품 방향), `/plan-eng-review`(아키텍처 잠금), `/review`(Staff Engineer 관점 코드 리뷰), `/qa`(Playwright 브라우저 자동화), `/ship`(PR 자동 생성), `/security`(OWASP+STRIDE 감사) 등입니다.

## Superpowers 개요

Jesse Vincent가 2025년 10월에 만들고 Anthropic 마켓플레이스에 등재된 오픈소스 에이전틱 스킬 프레임워크입니다. AI 코딩 에이전트에게 비타협적인 워크플로우를 강제합니다 — 코드 전에 설계, 기능 전에 테스트, 모든 작업 사이에 구조화된 리뷰.

Superpowers의 핵심 통찰은 클로드 코드의 기본적인 "바로 코딩 시작" 본능을 가로채고, 이를 먼저 적절한 엔지니어링 단계로 라우팅하는 스킬을 통해 이 문제를 해결할 수 있다는 것입니다. 5단계(Clarify → Design → Plan → Code → Verify)는 강제적이며, 각 단계에는 다음으로 넘어가기 전에 통과해야 하는 품질 게이트가 있습니다.

1~3단계는 저렴하고(텍스트 계획만), 4단계가 비싼데(실제 코드 작성), 계획이 이미 잠겨있기 때문에 4단계에서 첫 번째 시도에 성공률이 높아집니다.

## Harness 엔지니어링 도입 시 장점

**아키텍처·코드 품질 일관성** — gstack의 `/plan-eng-review`는 아키텍처 결정을 코드 작성 전에 잠그고, Superpowers의 Design/Plan 단계가 이를 보완합니다. 팀이 바뀌어도 동일한 구조적 기준이 유지됩니다.

**릴리스 파이프라인 가속화** — gstack의 `/ship`이 브랜치 동기화, 테스트, PR 생성을 자동화하며, Superpowers의 Verify 단계가 CI 게이트 역할을 합니다. `/autoplan` 커맨드는 CEO 리뷰, 디자인 리뷰, 엔지니어링 리뷰를 단일 파이프라인으로 연결합니다.

**보안 내재화** — gstack이 매 릴리스마다 OWASP/STRIDE 감사를 실행하고, Superpowers는 TDD를 워크플로우에 강제로 내장합니다.

**소규모 팀의 대형 팀 효과** — Garry Tan 본인의 사례로, YC를 운영하면서 파트타임으로 60일간 3개 프로덕션 서비스와 40개 이상의 기능을 출시했습니다.

**두 도구 병행 사용 전략** — gstack이 역할 자유도(필요할 때 원하는 역할 호출)에서 강하고, Superpowers가 프로세스 강제성(건너뛰기 불가한 단계)에서 강하기 때문에, 둘을 함께 도입하면 자유로운 병렬 개발과 엄격한 품질 게이트를 동시에 얻을 수 있습니다.

# 🦸 Superpowers (초능력)

**실제로 작동하는 에이전틱 기술 프레임워크 및 소프트웨어 개발 방법론**

## 📝 개요

Superpowers는 코딩 에이전트를 위한 완성된 소프트웨어 개발 방법론입니다. 조합 가능한 **기술(Skills)** 세트와 에이전트가 이를 반드시 사용하도록 만드는 초기 지침(Instructions)을 기반으로 구축되었습니다.

단순히 코드를 빠르게 생성하는 것이 아니라, AI 에이전트에게 실제 소프트웨어 엔지니어링 워크플로우를 부여하여 더 똑똑하고 안정적으로 작동하게 만듭니다.

---

## ⚙️ 작동 방식 (How it works)

코딩 에이전트를 실행하는 순간부터 작동합니다. 무작정 코드를 쓰는 대신, 한 걸음 물러나 사용자가 진정으로 하려는 작업이 무엇인지 묻는 것부터 시작합니다.

### 🔄 기본 워크플로우

1. **브레인스토밍 (brainstorming):** 코드를 쓰기 전 활성화됩니다. 질문을 통해 거친 아이디어를 다듬고, 대안을 탐색하며, 검증을 위해 설계를 섹션별로 제시합니다. 설계 문서를 저장합니다.
    
2. **Git 워크트리 활용 (using-git-worktrees):** 설계 승인 후 활성화됩니다. 새 브랜치에서 격리된 작업 공간을 만들고, 프로젝트를 설정하며, 깨끗한 테스트 베이스라인을 확인합니다.
    
3. **계획 수립 (writing-plans):** 승인된 설계를 바탕으로 작업을 2~5분 단위의 작은 태스크로 나눕니다. 각 태스크는 정확한 파일 경로, 완성된 코드, 검증 단계를 포함합니다.
    
4. **실행 (executing-plans):** 태스크별로 새로운 서브 에이전트(Subagent)를 파견하여 2단계 리뷰(사양 준수 확인 후 코드 품질 검토)를 거치거나, 사람이 체크포인트를 확인하며 일괄 실행합니다.
    
5. **테스트 주도 개발 (test-driven-development):** 구현 중에 활성화됩니다. **RED-GREEN-REFACTOR** 원칙을 강제합니다: 실패하는 테스트 작성 → 실패 확인 → 최소한의 코드 작성 → 통과 확인 → 커밋. 테스트 전에 작성된 코드는 삭제합니다.
    
6. **코드 리뷰 요청 (requesting-code-review):** 태스크 사이에 활성화됩니다. 계획 대비 리뷰를 수행하고 이슈를 심각도별로 보고합니다. 치명적인 이슈는 진행을 차단합니다.
    
7. **개발 브랜치 마무리 (finishing-a-development-branch):** 모든 태스크 완료 시 활성화됩니다. 테스트를 최종 확인하고 병합(Merge)/PR/유지/폐기 옵션을 제시한 후 워크트리를 정리합니다.

## 📚 기술 라이브러리 (Skills Library)

- **systematic-debugging:** 4단계 근본 원인 분석 프로세스.
    
- **verification-before-completion:** 문제가 실제로 해결되었는지 확실히 확인.
    
- **test-driven-development:** 항상 테스트를 먼저 작성하는 주기.
    
- **writing-skills:** 베스트 프랙티스에 따라 새로운 기술 생성.

## 🧠 철학 (Philosophy)

- **테스트 주도 개발 (TDD):** 언제나 테스트를 먼저 작성합니다.
    
- **임기응변보다 체계 (Systematic over ad-hoc):** 추측보다 프로세스를 중시합니다.
    
- **복잡성 감소 (Complexity reduction):** 단순함을 일차적인 목표로 삼습니다.
    
- **주장보다 증거 (Evidence over claims):** 성공을 선언하기 전에 반드시 검증합니다.
